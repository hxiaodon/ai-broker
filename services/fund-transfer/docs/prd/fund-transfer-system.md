# 出入金系统（Fund Transfer System）— Domain PRD

> **版本**：v1.0

## 一、业务概述

### 1.1 核心职责

出入金系统负责用户资金在**银行账户与券商托管账户**之间的双向流转，包括：

| 职能 | 说明 |
|------|------|
| **Deposit（入金）** | 用户从自己的银行账户向券商账户入金（ACH/Wire/FPS） |
| **Withdrawal（出金）** | 用户从券商账户向自己的银行账户出金（ACH/Wire/FPS） |
| **Bank Account Management** | 绑定、验证、管理用户的银行账户（最多 5 张） |
| **Compliance & Screening** | AML/KYC 审查、CTR/SAR 申报、Travel Rule 遵守 |
| **Ledger & Accounting** | 双分录账本、余额计算、对账 |
| **Reconciliation** | 实时/每日/每月的三方对账（内部账本 ↔ 银行回单 ↔ 托管行） |

### 1.2 支持的渠道与时间

| 方式 | 币种 | 手续费 | 预计到账 | 上线 |
|------|------|--------|---------|------|
| **ACH 转账** | USD | 免费 | 3-5 工作日 | Phase 1 ✅ |
| **Wire 电汇** | USD | $25/笔 | 当日（Fedwire cutoff **17:00 ET**；OFAC 合规截止 17:00 ET；各银行内部受理截止通常 14:00–16:00 ET，以合作银行协议为准） | Phase 1 ✅ |
| **FPS** | HKD | 免费 | 实时/同日 | Phase 2 |
| **CHATS** | HKD | 免费 | 同日 | Phase 2 |

## 二、核心业务规则

### 2.1 同名账户原则（Rule 1）

**定义**：用户**只能**向自己名下的银行账户入金或出金。

**规则**：

```
入金账户持有人姓名 ≡ KYC 认证的法定姓名  ✓
出金账户持有人姓名 ≡ KYC 认证的法定姓名  ✓
```

**执行**：

- ✅ 绑卡时：系统自动填充 KYC 姓名，用户只需验证；系统自动进行**模糊匹配**（忽略大小写、标点、前后空格）
- ✅ 出金时：出金账户下拉仅显示**已验证且姓名匹配**的账户
- ❌ 禁止：向第三方账户入金（如配偶、亲戚的账户）
- ✅ 例外：联名账户（Joint Account）中用户是命名持有人时可允许（需法务逐案确认）

**风险处理**：

- 如果绑卡时检测到**姓名不匹配**，系统在验证流程中**阻断**，提示"账户持有人姓名与 KYC 认证不一致"
- 如果出金时用户试图向非自己名下账户出金，系统拒绝，错误码 `WITHDRAWAL_ACCOUNT_MISMATCH`

### 2.2 可提现金额计算（Rule 4）

**定义**：在任何时刻，用户的**可提现金额**由以下因素组成：

```
可提现金额 = 总现金余额
           - 待结算资金（未到T+1/T+2）
           - 冻结中的出金申请金额
           - 保证金占用（Phase 1 = 0，后续支持 Margin）
           - Pending Deposits（等待银行确认）
```

**结算周期（与 Trading 系统契约）**：

| 市场 | 资产 | 结算周期 | 实施日期 |
|------|------|---------|---------|
| 美股 | US Stocks | T+1（Settlement Date = Trade Date + 1） | 2024-05-28 起 |
| 港股 | HK Stocks | T+2（Settlement Date = Trade Date + 2） | Phase 2 |

**具体计算逻辑**：

1. **卖出后资金**：
   - 用户卖出 100 股 AAPL，成交价 $150，成交金额 $15,000
   - Trade Date = 2026-03-29（周一）
   - Settlement Date = 2026-03-30（周二）
   - 2026-03-30 00:00 UTC 之前：$15,000 属于**待结算资金**，不可提现
   - 2026-03-30 00:00 UTC 之后：$15,000 变为**已结算现金**，可以提现

2. **现金入金**：
   - 用户通过 ACH 入金 $10,000
   - 银行确认到账后，**立即**可用于交易和出金（不需要等待额外结算）

3. **进行中的出金**：
   - 用户提交出金申请 $5,000，等待审批中
   - 该 $5,000 被**冻结**，不可再次出金或用于交易

**显示规范**（与 Mobile 契约）**：

Mobile 应同时展示：
- ✅ **账户总资产**：现金 + 持仓市值
- ✅ **可用现金**：今天可立即用于交易的金额
- ✅ **待结算资金**：T+1 未到的卖出收益（需注明"明天可提现"）
- ✅ **可提现金额**：可立即出金的金额（= 可用现金 - 冻结出金）

### 2.3 银行卡绑定与冷却期（Rule 1 + Rule 5）

**流程**：

```
绑定申请 → 微存款验证（1-3 天） → 验证成功 → 冷却期（3 天） → 可使用
```

**冷却期规则**：

| 绑定天数 | 入金 | 出金 | 说明 |
|---------|------|------|------|
| **0–3 天（冷却期）** | ❌ 禁止 | ❌ 禁止 | 新绑卡，风险等级最高 |
| **3–7 天** | ✅ 允许 | ⚠️ 可能触发人工审核 | 有效期内，但可能要求额外确认 |
| **≥ 7 天** | ✅ 正常 | ✅ 正常 | 充分验证，无额外限制 |

**异常处理**：

- 如果用户在冷却期内试图入金/出金，API 返回错误 `BANK_ACCOUNT_COOLDOWN`，Mobile 显示"请等待 X 天后再试"
- 微存款验证**超时 10 个营业日**未确认：该绑卡作废，需删除后重新绑卡（NACHA 规范；10 营业日 ≈ 14 日历日，但以营业日计以对齐 ACH 处理周期）

**冷却期叠加规则（SSOT）**：

存在两套独立的冷却限制，叠加时取**最严格约束**（两个限制同时满足才可操作）：

| 冷却类型 | 期限 | 限制范围 | 触发条件 |
|---------|------|---------|---------|
| **银行卡冷却期** | 绑卡后 3 天（0–3 天禁止） | 出金 + 入金 | 新绑定任意银行卡时 |
| **账户新用户冷却期** | 账户开立后 30 天 | 出金单笔 ≤ $1,000，只能出至主账户 | 账户注册后自动触发 |

叠加示例：

```
用户开户第 2 天，绑卡第 1 天：
  银行卡冷却: 禁止出金  ← 更严
  账户冷却:   出金≤$1K   
  实际限制:   禁止出金（取严）

用户开户第 5 天，绑卡第 5 天：
  银行卡冷却: 可能触发人工审核（3–7天区间）
  账户冷却:   出金≤$1K，只能出至主账户  ← 更严
  实际限制:   出金≤$1K + 只能出至主账户 + 人工审核（全部叠加）

用户开户第 35 天，绑卡第 10 天：
  银行卡冷却: 无限制（≥7天）
  账户冷却:   无限制（≥30天）
  实际限制:   无额外限制，按 KYC 正常限额
```

实现：限额校验函数需同时计算两个维度，返回最终约束集合；不得因"其中一个已过期"而跳过另一个的校验。

### 2.4 出金审批规则（Rule 5）

**决策树**：

```
用户提交出金申请
  ├─ 【Step 1】基础校验
  │   ├─ 金额 ≤ 0？→ 拒绝（INVALID_AMOUNT）
  │   ├─ 金额 > 可提现金额？→ 拒绝（INSUFFICIENT_BALANCE）
  │   └─ 账户被锁定？→ 拒绝（ACCOUNT_LOCKED）
  │
  ├─ 【Step 2】同名账户校验（Rule 1）
  │   └─ 出金账户 ≠ KYC 名下？→ 拒绝（ACCOUNT_MISMATCH）
  │
  ├─ 【Step 3】AML 筛查（Rule 2）
  │   ├─ 用户/银行在 OFAC SDN 中？→ 拒绝（AML_BLOCKED）
  │   ├─ 返回 REVIEW？→ 进入人工审核
  │   └─ 返回 PASS？→ 继续
  │
  ├─ 【Step 4】审批规则判断（下方矩阵）
  │   ├─ 满足自动审批条件？→ **自动审批**
  │   ├─ 满足人工审核条件？→ **进入人工审核队列**（1 工作日）
  │   └─ 大额出金？→ **合规专员审批**
  │
  └─ 【Step 5】提交至银行处理
      ├─ 银行成功？→ 出金完成，推送通知
      └─ 银行拒绝？→ 退款至账户，通知用户
```

**详细的审批矩阵**：

#### 自动审批（3 个条件**全部满足**）

| 条件 | 说明 | 备注 |
|------|------|------|
| ✅ 金额 ≤ 日限额 | 金额在 KYC Tier 的日限额范围内 | 见下方 KYC 限额表 |
| ✅ 银行卡验证 > 3 天 | 绑卡时间已超过冷却期 | 降低新卡欺诈风险 |
| ✅ 无 AML 标记 | 用户和银行都未被标记为可疑 | AML 结果 = PASS |
| ✅ 风险评分 = LOW | 系统欺诈检测评分为低 | 实时计算，见 Rule 2 |
| ✅ 有历史出金记录 | 用户曾成功出金过至少 1 次 | 防止首次大额出金欺诈 |

**自动审批 SLA**：< 1 分钟

#### 人工审核（满足**任意一个**条件）

| 条件 | 触发阈值 | 处理 |
|------|---------|------|
| 金额过大 | > $50,000 USD 单笔 | 人工审核（1 工作日） |
| 日累计额度 | > 日限额 80% | 人工审核（1 工作日） |
| 冷却期不足 | 3 天 ≤ 绑卡时间 < 7 天 | 人工审核（建议增强验证） |
| 账户新开 | 注册时间 < 30 天 | 人工审核 |
| 风险评分中等 | MEDIUM or HIGH | 人工审核 |
| AML 需人工审查 | AML 返回 REVIEW | 人工审核 |

**人工审核 SLA**：**4 工作小时**内完成（超时自动升级为 L2，同时通知合规主管）

#### 合规专员审批（需要升级）

| 条件 | 处理 |
|------|------|
| 大额出金 > $200,000 USD | 合规专员审批 |
| 触发 SAR（Suspicious Activity Report） | 合规专员审批 |
| 多次 AML 筛查失败（30 天内 ≥ 3 次） | 合规专员审批 |
| 用户在内部风险名单上 | 合规专员审批 |

**合规审批 SLA**：1-2 个工作日（可能需要与用户沟通）

### 2.5 KYC 等级限额(命名 + 数值 SSOT)

> ⚠️ **SSOT 公告**: KYC tier 命名和具体限额数值的**唯一权威来源是 AMS 服务**(`AMS.GetAccountKYCTier(user_id)` 返回的 `tier_id` + `limits` 结构体)。本文表格仅作 PRD 阶段示例;**实际生产值以 AMS 服务契约 `docs/contracts/ams-to-fund.md` 为准**,且要求与 `operations-and-edge-cases.md` §2.1、`fx-conversion-flow.md` §6.1 一致。

**四档体系**(从 2026-05 起替换原 Tier 1/2/3 命名):

| KYC 等级 | 适用人群 | 入金单笔/日 | 入金月限 | 出金单笔/日 | 出金月限 | 说明 |
|---------|----------|------------|---------|------------|---------|------|
| **Basic** | KYC 提交待审 / 初级身份验证 | $2,000 | $10,000 | $1,000 | $5,000 | 入金保守低于 CTR 阈值;无历史出金记录用户默认 |
| **Standard** | 完整 KYC(身份+银行+SSN/HKID) | $50,000 | $200,000 | $25,000 | $100,000 | 大多数活跃用户 |
| **Enhanced** | 完整 KYC + 收入证明 + RM 复核 | $500,000 | $2,000,000 | $250,000 | $1,000,000 | 高净值客户 |
| **VIP** | 合规专员逐案审核 | 无固定上限 | 无固定上限 | 无固定上限 | 无固定上限 | > $200K 单笔触发合规升级,所有交易均要人工审批 |

**实现**:

- 每次入金/出金/换汇前,fund-transfer 调用 AMS:`GetAccountKYCTier(user_id)` → 返回 `{tier_id, limits, valid_until}`
- 比较用户申请金额与 Tier 限额,超过则拒绝或触发人工审核(走 §2.4 审批矩阵)
- AMS 不可用时降级策略见 `operations-and-edge-cases.md` §2.6 — **缓存失效时一律拒绝出入金**(P0-5 修正后规则)

### 2.6 AML 筛查与 CTR 申报（Rule 2）

**执行时点**：

- 每笔入金或出金都**必须**进行 AML 筛查，**不管金额大小**
- **FX 换汇、内部账户转账、退款**等所有资金移动事件同样必须经过 AML 筛查;不存在"入金时已筛查所以后续可跳过"的例外(OFAC 名单每周更新数次,筛查结果只对当次事件有效)
- 筛查在 API 层**同步执行**（不异步），超时 SLA = 3 秒

**筛查范围**：

```
筛查对象：
├─ 用户法定姓名 + DOB
├─ 用户国籍/住所国
├─ 银行名称
├─ 银行国家
├─ SWIFT Code（国际转账时）
└─ 受益人姓名(对外汇款时)

筛查列表（美国）：
├─ OFAC SDN（Specially Designated Nationals）
├─ OFAC Sectoral Sanctions（特定国家/行业）
├─ OFAC Non-SDN Consolidated List
├─ Foreign Sanctions Evaders List
└─ Non-SDN Menu-Based Sanctions List

筛查列表（港澳）：
├─ SFC 指定人员/实体列表
├─ AMLO Part 4A 指定名单
└─ 联合财智组（JFIU）可疑交易名单
```

**OFAC 50% Rule(强制要求)**:

被一个或多个 SDN 实体**直接或间接合计持股 ≥ 50%** 的公司,即使不在任何 OFAC 名单上,也视同被制裁。实现时:

- AML 筛查必须支持 Ultimate Beneficial Owner(UBO)追溯至少 **2 层**股权
- 商业 SaaS 供应商必须支持 UBO 数据库(推荐 Refinitiv World-Check / ComplyAdvantage / LexisNexis Bridger,详见 `docs/references/aml-screening-vendors.md`)
- 命中 50% Rule 视同 SDN BLOCK,流程同 BLOCK 结果
- 单案历史罚款记录达 $1.2 亿(2019 OFAC Settlement),不可忽视

**筛查结果**：

| 结果 | 含义 | 处理 |
|------|------|------|
| **PASS** | 无风险，直接通过 | 继续处理 |
| **REVIEW** | 需人工审查（模糊匹配、相似名字、50% Rule 中等置信度等） | 进入人工审核队列（待合规人员确认） |
| **BLOCK** | 确认在黑名单上 或 50% Rule 命中 | 拒绝交易，即刻冻结账户，通知合规部门;**对用户使用脱敏话术,不得透露 AML 原因** |

**CTR 自动申报(Rule 2)**:

当入金或出金**金额严格大于 CTR 阈值**(`> $10,000`,不含等于,FinCEN "in excess of")时,系统自动生成并提交 CTR:

| 地区 | 阈值 | 管理部门 | 提交时间 |
|------|------|---------|---------|
| **美国** | `> $10,000 USD` | FinCEN | 交易后 15 个自然日内 |
| **香港** | `> HK$120,000` | JFIU | 交易后 10 个工作日内 |

**CTR 内容**：涉及用户身份、银行信息、交易金额、交易类型、交易日期等。USD 与 HKD 不做跨币种合并;同日累计 / 跨账户聚合按 `.claude/rules/fund-transfer-compliance.md` Rule 2 SSOT 定义。

**分拆交易检测(Rule 2,双窗口)**：

SSOT 在 compliance Rule 2;PRD 简述如下,实现细节见 `specs/operations-and-edge-cases.md` §3.2:

| 窗口 | 触发条件 |
|------|---------|
| **24h 滑动** | 同用户同方向同币种 ≥ 3 笔,单笔均 < CTR 阈值,累计 ≥ $8,000 USD(HK$96K) — intra-day 拆分 |
| **7d 滚动**(FinCEN 标准) | 同条件 ≥ 3 笔,累计 ≥ $10,000 USD(HK$120K) — 跨日拆分 |

聚合维度:跨渠道(ACH/Wire/FX-funded)、跨账户(同 SSN/HKID)。任一命中:

- 当前请求标记 `AML_STATUS = REVIEW`(不阻断,除非风险评分 HIGH)
- 关联历史成功交易纳入 SAR 评估
- 生成 SAR 草稿,合规官 30 日内决定是否上报(31 CFR 1023.320)
- **严禁通知用户**(Tipping-off 违反 31 U.S.C. § 5318(g)(2)/AMLO §25A(5))
- 如同时触发 CTR,CTR 与 SAR **独立流程独立审批**,客户通信统一使用合规审查脱敏话术,不得提及 CTR/SAR/AML 任何字眼

### 2.7 Travel Rule 合规（Rule 3）

**定义**：对于**跨境或金额较大**的转账，金融机构必须传输发款人和收款人信息给对方。

**触发条件**：

```
Transfer Amount > $3,000 USD  或  Transfer Amount > HK$8,000 HKD
```

**必传信息**：

**发款人信息**：
- 姓名（与身份证件一致）
- 账号（最后 4 位）
- 住所地址

**收款人信息**：
- 姓名（银行账户持有人）
- 账号（最后 4 位）
- 住所地址（如可获得）

**实现**：

1. 用户提交出金申请 > $3,000 USD
2. 系统自动收集上述信息
3. 通过 SWIFT/FedNow 等渠道将信息附加到汇款指令中
4. 接收行验证收款人信息，如不符可拒绝接收或返还

**记录保留**（Rule 9）：
- Travel Rule 相关记录保留最少 **5 年**
- 包含：发款人信息、收款人信息、传输日期、传输方式

### 2.8 Ledger 完整性（Rule 6）

**原则**：每笔资金移动都必须遵循**双分录（Double-Entry Bookkeeping）**原则。

**基本规则**：

```
每笔交易 → 对账分录（Debit Entry）+ 贷账分录（Credit Entry）

例：用户入金 $1000
┌─────────────────────────────────────────┐
│ Debit Entry:  用户账户 +$1000           │
│ Credit Entry: 银行待清队列 -$1000      │
│ 日期：2026-03-29 14:30 UTC              │
│ Transaction ID：txn_001                 │
└─────────────────────────────────────────┘
```

**账本属性**：

| 属性 | 说明 | 实现 |
|------|------|------|
| **Append-Only** | 分录**永不修改或删除** | INSERT-only 表结构，无 UPDATE 操作 |
| **Immutable** | 历史记录完全追溯 | 所有分录带时间戳、user_id、tx_id |
| **Sum Invariant** | 所有用户余额之和 = 平台托管账户余额 | 每日对账程序验证此不等式 |

**纠错机制**：

如果发现分录有误（如重复计账），**不是修改原分录**，而是生成**反向分录**：

```
原分录（错误）：
Debit:  User A +$1000
Credit: Bank Queue -$1000

反向分录（纠错）：
Debit:  Bank Queue +$1000
Credit: User A -$1000

最终效果：User A 回到原始余额
```

**对账验证**（Rule 6）：

系统每日自动进行三方对账：

```
内部账本（Ledger 表）
    ↓ match ↓
银行回单（Bank Statement）
    ↓ match ↓
托管行报告（Custodian Balance Report）
```

- ✅ 完全匹配 → 放行
- ⚠️ 差异 < $0.01 → 记录待查
- ❌ 差异 ≥ $0.01 → 自动告警，暂停相关账户的出金操作
- ❌ 差异 > $100 → 立即上报合规部门，可能冻结账户

## 三、入金流程（Deposit Flow）

### 3.1 入金决策树

```
用户发起入金申请
  │
  ├─ 【Step 1】基础校验
  │   ├─ 金额有效性？
  │   ├─ 金额 ≤ Tier 日限额？
  │   └─ 选中的银行卡已验证且非冷却期？
  │
  ├─ 【Step 2】AML 筛查（同步）
  │   ├─ 用户在黑名单？→ 拒绝
  │   ├─ 银行在黑名单？→ 拒绝
  │   └─ 需人工审查？→ 进入人工队列（继续处理，但标记为待审）
  │
  ├─ 【Step 3】入金额度检查
  │   └─ 单笔金额 ≤ 日限额 ≤ 月限额？→ 继续
  │
  ├─ 【Step 4】微存款验证
  │   └─ 银行卡已验证？→ 继续；否则拒绝
  │
  ├─ 【Step 5】生成入金指令
  │   ├─ 向银行发送 ACH/Wire 指令
  │   ├─ 记录 Bank Reference ID（幂等性标识）
  │   └─ 状态 = PENDING_BANK_CONFIRMATION
  │
  ├─ 【Step 6】等待银行回调
  │   ├─ 银行确认成功？→ 余额立即增加，推送通知
  │   ├─ 银行拒绝？→ 记录拒绝原因，通知用户
  │   └─ 银行超时（5 天）？→ 标记为 TIMEOUT，定期查询状态
  │
  └─ 【Step 7】AML 人工审查（如 Step 2 返回 REVIEW）
      ├─ 合规人员审查 → 确认无风险 → 继续 Step 5
      └─ 合规人员审查 → 确认可疑 → 拒绝，通知用户
```

### 3.2 入金方式与 SLA

| 方式 | 描述 | 费用 | 到账时间 | 使用场景 |
|------|------|------|---------|---------|
| **ACH** | 美国清算所电子转账 | 免费 | 3-5 工作日 | 日常入金，金额不急 |
| **Wire** | 银行电汇（SWIFT） | $25/笔 | 工作日当日（14:00 ET 前） | 大额或紧急入金 |

### 3.3 入金后的资金状态

用户成功入金后，资金流转如下：

```
银行确认 → 托管行账户收到 → Fund Service 记账 → 用户可见
  ↓            ↓                   ↓              ↓
 T            T                  T              T
 (银行处理)  (托管行，1-3 天)   (记账，实时)   (余额+1，立即可用)
```

**关键承诺**：
- ✅ 入金成功率 ≥ 97%（银行确认率）
- ✅ 成功入金后**立即可用**（可交易、可再次出金）
- ✅ 入金过程中，所有失败都会通知用户，可重试

### 4.3 出金被银行退回的处理

**场景**：出金申请已被平台受理并提交至银行，但银行随后将资金退回（如收款账户已关闭、账户信息错误等）。

**完整处理流程**：

```
出金资金汇出至银行
  ↓
  ├─ 银行成功入账 → 出金完成，推送通知
  └─ 银行拒绝处理，退回资金（3-5 个工作日）
      ├─ 资金返还至证券账户
      ├─ 账户余额**立即恢复**
      ├─ 系统推送退款通知
      ├─ App 展示退回原因（银行提供）
      └─ 根据退回原因引导用户
          ├─ 账户信息错误 → 引导检查和修正
          ├─ 账户已关闭 → 引导删除失效卡，重新绑定
          └─ 其他银行原因 → 建议联系客服或收款银行
```

**关键承诺**：
- ✅ 退款到账后**立即可用**
- ✅ 用户可立即重新出金或交易
- ✅ 完整的退款原因说明

### Rule 8：幂等性（Idempotency for Fund Operations）

| 维度 | 说明 |
|------|------|
| **PRD 位置** | § 4.1 入金流程 / § 4.2 出金流程（隐含） |
| **业务承诺** | 每笔请求含 UUID Idempotency-Key；系统 72 小时缓存 |

| 维度 | 说明 |

### Rule 10：错误处理（Error Handling for Fund Operations）

| 维度 | 说明 |
|------|------|
| **PRD 位置** | § 4.3 出金被银行退回的处理 + Mobile PRD-05 § 10 异常边界 |

如平台向客户支付现金余额利息（Money Market / Cash Sweep）：
- 每位 US Person 全年累计利息 **≥ $10** 需出具 1099-INT
- 截止日：对客户 1 月 31 日，对 IRS 2 月 28 日 / 3 月 31 日

> 若 Phase 1 不提供利息产品，此条可暂缓至 Phase 2 启用时处理。

---

### 13.4 Form 1042-S — 非美国人士预扣税申报

**适用**：向 Non-Resident Alien (NRA) 用户支付的以下美国来源所得：
- 美股股息 (US Dividends)
- 债券利息
- 美股期权权利金（如适用）

**预扣税率**：
- 默认 **30%**（IRC §1441，用于 US-source income 对非居民）
- 可享受税收协定优惠税率（中国大陆居民 → 美国股息协定税率 **10%**；香港目前无 US-HK 税收协定，仍 30%）
- 用户必须通过 W-8BEN 申请协定税率，并填写申请国和 TIN

**执行时点**：Trading Engine 处理股息分配时，fund-transfer 同步执行预扣并写入 `withholding_log` 表；净股息（税后）入账到用户余额。

**申报截止日**：对 IRS，**3 月 15 日**；对用户，**3 月 15 日**。

---

### 13.5 FATCA 合规（Foreign Account Tax Compliance Act）

FATCA（IRC §§ 1471-1474）要求外国金融机构（FFI）向 IRS 报告美国账户持有人信息，否则承担 **30% FATCA 预扣税**。

**本平台适用场景**：
- 若平台在 HK 注册为 FFI：需与 IRS 签订 FFI Agreement（或依托 HK 的 IGA 协议）
- 识别美国账户：W-9 标识 US Person；有美国地址/电话/美国授权人的账户需进一步 due diligence
- 向 IRS 报告：IRS Form 8966（FATCA Report），年度申报，截止 **3 月 31 日**
- FATCA 预扣适用于"Withholdable Payments"——美国来源股息、利息等

**实施要先于 HK 市场上线**：需法务完成 HK IGA 适用性分析，决定以 IGA Model 1 (报告)还是 Model 2 (直接 FFI Agreement) 方式合规。

---

### 13.6 备用预扣税（Backup Withholding）

当 US Person 未提供有效 W-9 或 TIN，或 IRS 通知 TIN 不匹配时，需对：
- 股息、利息、1099-B 所报告的收益
- 实施 **28%** 备用预扣税，汇缴至 IRS（Form 945 年度汇缴）

平台在用户 KYC 时必须验证 TIN 格式（SSN: 9 位，EIN: 9 位），并留存 IRS TIN Matching 响应日志。

## 十四、账户注销与休眠账户管理（P1-8）

### 14.1 用户主动注销

**前提条件（全部满足才允许注销）**：

- 无 pending 订单（无未成交、部分成交的委托）
- 无未结算持仓（证券持仓清零，或已完成 T+1/T+2 结算）
- 账户余额 ≥ 0（不存在负余额）
- 无进行中的 AML 调查
- 无悬挂资金（suspense_funds 表中无属于该用户的未对账记录）

**余额清退流程**：

```
用户申请注销
  │
  ├─ 余额 = 0 → 直接注销（跳过清退）
  └─ 余额 > 0 → 触发清退出金
       │
       ├─ 自动发起"清退出金"至用户的默认同名银行账户（走正常出金流程、AML 筛查、审批矩阵）
       ├─ 清退出金完成（银行确认到账）→ 账户标记 CLOSED
       └─ 清退出金失败（如银行账户已失效）→ 暂停注销，通知用户更换银行账户后重试
```

**注销后数据留存（合规留存，非删除）**：

| 数据类型 | 留存期限 | 依据 |
|---------|---------|------|
| KYC / 身份文件 | 账户关闭后 6 年 | KYC 要求 |
| AML 筛查结果 | 7 年 | BSA / AMLO |
| 出入金记录 | 7 年 | SEC 17a-4 / SFO |
| 账本分录 | 无限期 | 业务连续性 |
| CTR / SAR 申报记录 | 5 年 | FinCEN / JFIU |

> 重要：账户注销只是将账户状态改为 CLOSED，所有合规数据按上表继续保留，不执行物理删除。合规团队在留存期内仍可查询。

**特殊阻断场景**：

- 如有 AML 调查未结：账户维持 FROZEN 状态，禁止注销，直至调查完成（合规团队解除后才可继续注销流程）
- 如有悬挂资金未对账完毕：禁止注销，suspense_funds 处理完毕（资金匹配或上报）后方可注销

---

### 14.2 休眠账户（Dormant Account）管理

**定义**：连续 3 年（1095 天）无任何账户活动，包括：无登录、无交易、无出入金、无任何 API 请求。

**适用法规**：
- **FINRA Rule 4530**：要求识别和报告客户投诉及不规则事件
- **各州 Unclaimed Property Act**（Escheatment）：3–5 年无活动的资产须上报州政府

**休眠触发与处理流程**：

| 阶段 | 触发条件 | 平台动作 |
|------|---------|---------|
| **警告 1** | 连续 1 年无活动 | 发送"账户即将休眠"警告邮件，告知后续影响 |
| **警告 2** | 连续 2 年无活动 | 再次警告，更换通知方式（短信 + 推送），要求用户登录确认 |
| **休眠标记** | 连续 3 年无活动 | 账户状态改为 DORMANT，限制交易，仅允许提现和账户重激活操作 |
| **Escheatment** | 连续 5 年（或州法规定期限）无活动 | 启动 Escheatment 流程（见下方） |

**Escheatment 执行流程**：

```
满足 Escheatment 触发期限
  │
  ├─ 提前 6 个月：发送最终通知（邮件 + 挂号信至 KYC 登记地址），告知将上报州政府
  ├─ 提前 3 个月：若仍无响应，合规专员审批启动 Escheatment
  ├─ 执行：向用户注册地所在州 Unclaimed Property 部门申报（Form UP-1 或州指定表格）
  ├─ 汇款：将余额汇至该州 Unclaimed Property 基金
  └─ 记录：账户标记 ESCHEATED，写入 audit_log，保留完整申报记录
```

**账户重激活**：

- 用户可随时联系客服或在 App 申请重激活
- 需完成身份验证（再次 KYC 或生物识别确认）
- 如余额已 Escheated：系统告知用户向对应州政府申请余额取回（提供州 Unclaimed Property 查询链接）；重激活后账户余额为 $0，可重新入金

## 十五、非标准账户类型（P1-9）

### 15.1 联名账户（Joint Account）

**KYC 要求**：

- 所有联名持有人均须独立完成 KYC（每人提交身份文件、住址证明）
- 任意一名持有人被 AML 筛查 BLOCK，整个账户立即冻结，停止所有出入金和交易操作

**同名出金原则**：

- 出金银行账户的持有人，须为"联名账户的任意持有人之一"（不要求必须是本次操作的发起人）
- 姓名匹配时，对 `joint_account_holders` 表进行 OR 匹配，任一 holder 姓名匹配即通过同名校验

**签名权限模式**（开户时选择，不可在线变更，需合规审批）：

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **Either/Or** | 任意一名持有人可单独发起出入金和交易 | 夫妻共同账户等 |
| **Joint** | 需所有持有人通过各自账号授权后才能执行 | 合伙投资账户等 |

**继承处理**：

- 一方持有人去世：由存活方提供经认证的死亡证明
- 合规专员审核后，将签名权限切换为单人模式（Either/Or）
- 通知合规部门，写入 audit_log，保留死亡证明扫描件

---

### 15.2 未成年人账户（UTMA / UGMA Custodial Account）

**账户性质**：监护人（Custodian）为账户法律持有人，受益人（Minor）是未成年子女。账户归属于未成年人，由监护人代为管理。

**KYC 要求**：

- 监护人：完成完整成人 KYC（身份文件 + W-9/W-8BEN）
- 未成年人：仅需提供出生证明（不做独立 KYC，不收集 SSN/HKID）

**交易限制**（Phase 1）：

- 不允许融资交易（Margin）
- 不允许期权交易
- 可交易标的：普通股票（美股 / 港股，与标准账户相同）

**出入金规则**：

- 银行账户必须在监护人名下（同名原则以监护人 KYC 姓名为准）
- 出入金由监护人发起，经正常审批流程处理

**未成年人成年移交**：

- 系统根据出生证明中的出生日期自动计算成年日期（18 岁或 21 岁，取决于账户开立时所选的州法）
- 成年前 90 天：系统通知监护人，引导发起账户移交流程
- 移交流程：监护人与受益人（成年后）共同完成线上申请；受益人作为新账户持有人重新完成 KYC；原监护人权限撤销；账户类型从 UTMA/UGMA 改为标准个人账户

---

### 15.3 企业账户（Corporate Account）

**KYC 要求**：

- 实体 KYC：商业注册证、公司章程、营业执照（取决于注册地）
- UBO（Ultimate Beneficial Owner）KYC：持股比例 ≥ 25% 的最终受益人须全部完成个人 KYC

**AML 要求**：

- 所有 UBO 均须通过 OFAC SDN 筛查和 OFAC 50% Rule 检查
- FinCEN CDD Rule（Customer Due Diligence）要求：UBO 信息须在开户时收集并上报
- 企业注册地为高风险国家/地区时，触发 Enhanced Due Diligence（EDD）

**出入金规则**：

- 银行账户须为企业对公账户（账户持有人为企业名称，不允许以个人账户代替）
- 同名校验时，银行账户持有人名称须与企业 KYC 登记名称匹配（同名原则以企业名义执行）

**限额与审批**：

- 企业账户享有 Enhanced 或 VIP 级别限额（从 AMS 获取，同个人账户 KYC 等级机制）
- 首次出入金须合规专员人工审批，建立企业账户风险档案

---

### 15.4 信托账户（Trust Account）

**KYC 要求**：

- 信托文件（Trust Deed / Trust Agreement）
- 受托人（Trustee）个人 KYC（完整成人 KYC）
- 受益人信息（姓名、关系、受益比例）——不要求受益人独立 KYC，但需存档

**出入金规则**：

- 以受托人名义发起出入金操作
- 银行账户须为信托账户（账户名称含"Trust"或"Trustee"字样，或出具银行证明）
- 同名校验：银行账户持有人须与受托人姓名或信托名称匹配

**受托人变更**：

- 必须提供法院认定文件（Court Order）或继任受托人任命文件
- 合规专员审核后更新受托人信息，旧受托人权限**立即撤销**
- 写入 audit_log，存档所有变更文件

---

### 15.5 继承账户（Deceased Account）

**触发条件**：收到经认证的死亡证明，或继承人/遗产执行人主动通知。

**立即响应**：

- 冻结所有交易和出入金操作（防止未授权访问）
- 已 pending 的订单：视情况保留（允许成交至结算完成），不强制平仓
- 发送确认通知至账户登记邮件（如有存活联系人）

**遗产处理流程**：

```
收到死亡证明及继承人联系
  │
  ├─ 遗嘱执行人（Executor）提供遗嘱认证文件（Letters Testamentary）
  │   或
  ├─ 遗产管理人（Administrator）提供遗产管理令（Letters of Administration）
  │
  ├─ 合规专员审核文件
  │
  ├─ 核实继承账户或指定出金账户
  │
  └─ 将余额清退至遗嘱指定账户或继承人账户（走正常清退出金流程，AML 对继承人筛查）
```

**数据合规留存**：

- 原账户所有 KYC、AML、出入金记录、账本分录均按 §14.1 留存规则保留 7 年，不因账户持有人去世而提前删除
- 继承过程中新产生的文件（死亡证明、法院文件）存入 WORM S3，与原账户档案关联

## 十六、Reg E 客户争议处理（P1-10）

> 本章覆盖美国《电子资金转账法》Regulation E（12 CFR Part 1005）对未授权 ACH 交易的客户保护义务。

### 16.1 适用范围

- **仅适用于**：美国消费者个人账户（US Person，非企业账户）
- **触发场景**：客户主张某笔 ACH 入金或出金是"未经本人授权"的转账

### 16.2 客户申诉时限

- 客户须在收到账单（或首次可查看该笔交易记录）后 **60 天内**提出书面争议（Reg E §1005.11）
- 超过 60 天：不再享有 Reg E 法定保护；平台可视具体情况酌情进行人道主义处理，但无法定义务

### 16.3 平台处理义务

| 时间节点 | 平台义务 |
|---------|---------|
| 收到争议（T+0） | 书面确认收到（邮件 + 站内信），提供参考编号（dispute_case_id） |
| T+10 工作日 | **发放临时信用（Provisional Credit）**：向客户账户暂时退回争议金额，使客户可正常使用资金 |
| T+45 日历日 | 完成调查并作出最终结论，书面通知客户 |
| 结论：确认未授权 | 将临时信用转为永久信用；向原发出方银行（ODFI）发起 ACH Return（Return Code R10 或 R11） |
| 结论：确认合规授权 | 撤销临时信用（从账户扣回）；书面向客户说明结论依据（不得透露 AML 相关信息） |

### 16.4 争议升级路径

```
客户通过 App / 客服热线提交争议
        │
        ▼
Ops 初审（SLA: T+1 工作日）
        │
        ├── 明显操作错误（金额/时间完全不符）
        │       → 直接受理，发起 ACH Return，无需等 T+10
        │
        └── 需进一步调查
                → 发出临时信用（Provisional Credit）
                → 启动正式调查
                        │
                        ▼
                T+45 日：
                ├── 联系 ODFI 核实授权记录
                ├── 查看平台 ACH 授权日志（用户点击同意时间戳、IP、设备指纹）
                └── 作出最终结论
                        │
                        ├── 确认未授权 → 永久退款，向 ODFI 发 Return，评估是否触发 SAR
                        └── 确认已授权 → 撤销临时信用，书面回复客户
```

### 16.5 内部数据要求

**ACH 授权记录保留**（争议举证材料）：

- 用户每次点击"同意 ACH 授权"时，系统须记录：操作时间戳（UTC）、用户 IP 地址、设备指纹（device_id）、授权协议版本号
- 记录保存 **6 年**（作为 Reg E 争议举证材料）

**争议状态机**：

争议记录存储在 `dispute_cases` 表，状态流转如下：

```
RECEIVED → PROVISIONAL_CREDIT_ISSUED → INVESTIGATING → RESOLVED
```

| 状态 | 说明 |
|------|------|
| RECEIVED | 争议已收到，待 Ops 初审 |
| PROVISIONAL_CREDIT_ISSUED | 临时信用已发放，调查进行中 |
| INVESTIGATING | 正在与 ODFI 及内部日志核实 |
| RESOLVED | 最终结论已作出，临时信用已转为永久或已撤销 |

## 十七、同名账户判定细则（P1-14）

> 本章是 §2.1 同名账户原则的补充细则，应与 §2.1 交叉引用。

### 17.1 姓名匹配规则

| 场景 | 处理方式 |
|------|---------|
| 完全一致（仅大小写或空格差异） | 自动通过 |
| 中文姓名 vs 英文拼音（如"张伟" vs "Zhang Wei"） | 自动通过（系统做中英文映射，基于 KYC 录入的双语姓名） |
| 中文姓名 vs 不同拼音方案（粤拼 vs 普通话拼音） | 人工审核 |
| 婚前姓 vs 婚后姓（附结婚证明） | 人工审核，审核通过后更新 KYC，更新前后均可使用 |
| 连字符 / 撇号（O'Brien vs OBrien；Mary-Jane vs Mary Jane） | 自动通过（忽略连字符和撇号再比较） |
| 缩写（Robert Smith vs R. Smith） | 自动拒绝，需提供完整姓名的银行账户 |
| 用户主张两个名字是同一人（提供证明文件） | 人工审核 |

### 17.2 模糊匹配算法

- **算法**：Jaro-Winkler 相似度（适合人名比对）

| 相似度区间 | 处置 |
|-----------|------|
| ≥ 0.95 | 自动通过 |
| 0.80 – 0.95 | 人工审核 |
| < 0.80 | 自动拒绝 |

- **中文姓名处理**：先转拼音（标准普通话拼音），再做 Jaro-Winkler 比对；同时与 KYC 库中录入的双语姓名映射表对照
- **阈值调整权限**：阈值可由合规团队在 Admin Panel 动态配置，每次变更须合规官审批并写入 audit_log

### 17.3 联名账户的出金同名校验

- 联名账户出金时，银行账户持有人为"联名账户的任意持有人之一"即视为同名通过
- 实现：同名校验逻辑对 `joint_account_holders` 表进行 OR 匹配，任一 holder 姓名满足匹配阈值即通过

### 17.4 人工审核升级路径

```
系统自动校验结果：相似度 0.80–0.95（不确定区间）
        │
        ▼
Ops 人工审核（SLA: 2 工作小时）
        │
        ├── 用户提交证明文件（护照、结婚证、改名证明、公证文件）
        │       → 审核通过
        │       → 更新 KYC 姓名记录（写入 audit_log）
        │       → 银行账户通过同名校验
        │
        └── 无法验证（文件不足或文件与申请不符）
                → 拒绝
                → 通知用户：请更换持有人与 KYC 姓名一致的银行账户
```
