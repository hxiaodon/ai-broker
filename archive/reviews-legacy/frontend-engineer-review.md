# 管理后台前端工程评审报告

**文档类型**: 前端工程评审（Admin Panel Frontend Engineering Review）
**评审对象**: 移动端 APP 设计方案 v1 / v2 / v3-supplement + UX 设计师审核问题清单
**评审视角**: Admin Panel Frontend Engineer（React 19 + TypeScript 5 + Ant Design Pro 6）
**评审日期**: 2026-03-11
**评审人**: Frontend Engineer (Claude)

---

## 执行摘要

移动端设计方案详尽地定义了 C 端用户界面，但**完全没有涵盖运营后台（Admin Panel）的功能需求**。对于一个受 SEC/FINRA/SFC 监管的跨境券商，Admin Panel 不是附加产品，而是**合规运营的基础设施**。

根据对三版设计文档的完整分析，管理后台至少需要 **7 大功能模块、约 40 个页面**，涵盖 KYC 审核、出入金管理、订单监控、合规报告等核心运营场景。本报告将系统性地推导这些需求，评估技术复杂度，并向产品和 UX 团队提出明确的补充要求。

**结论**: Admin Panel MVP 工期约 **16-20 周**（4-5 个月），建议与移动端并行立项，不能延后。

### 1.2 KYC 审核后台（Identity & Onboarding Review）

**背景**: v3 设计了 9 步 KYC 流程，其中人脸识别失败超 3 次会触发"人工视频审核"，OCR 失败会触发"人工审核"，审核结果分为通过/补材料/拒绝三种。这些人工节点必须有对应的后台界面。

**所需页面**:

#### 1.2.1 KYC 审核队列（主列表页）

```
功能要求:
- ProTable 展示待审核申请列表
- 列：申请编号、姓名、国籍、提交时间、KYC 类型（新开户/补材料/重新审核）
- 状态筛选：待审核 / 审核中 / 需补材料 / 已通过 / 已拒绝
- 优先级排序：等待时长（SLA 告警：>2 工作日变红）
- 快速搜索：手机号、邮箱、姓名、证件号（部分脱敏）
- 批量操作：批量分配审核员
- 实时计数 Badge：显示当前待审核数量（WebSocket 推送）
```

#### 1.2.2 KYC 详情审核页（核心页面）

```
功能要求:
- 左侧：用户提交的信息（个人信息/就业/财务/税务）
- 中央：证件图片查看器（支持旋转、缩放、对比）
  - 证件正面 + 反面并排展示
  - OCR 识别结果高亮标注（姓名、证件号、有效期）
  - 历史审核记录（同一用户之前的提交历史）
- 右侧：人脸识别比对结果（相似度分数、活体检测结果）
- 底部：审核操作区
  - [通过] [拒绝] [请求补材料]
  - 拒绝/补材料需选择原因（预设选项 + 自定义）
  - 所有操作需要审核员 ID 和时间戳（不可修改）
- 内嵌：AML 初步筛查结果展示（OFAC/制裁名单命中情况）
```

#### 1.2.3 KYC 进度追踪（Dashboard）

```
指标卡:
- 今日新增申请数
- 待审核队列长度
- 平均审核时长（目标 <4 小时）
- 自动审核通过率 vs 人工审核率
- 本月 KYC 通过率 / 拒绝率

图表:
- 按小时的新增申请量折线图（实时）
- 各拒绝原因分布饼图
- 审核员工作量对比柱状图
```

### 1.3 出入金管理（Fund Transfer Management）

**背景**: v3 出入金设计包括：ACH/Wire 入金、ACH/Wire 出金、换汇、银行卡绑定（Plaid/小额打款验证）、出金人工审核触发（>$50,000）。每个环节都需要后台对应操作。

**所需页面**:

#### 1.3.1 出入金概览 Dashboard

```
实时指标（WebSocket，刷新间隔 5 秒）:
- 今日入金总额（USD + HKD 分开）
- 今日出金总额
- 待处理出金笔数（需人工审核）
- 资金对账状态（绿色正常 / 红色异常）
- 平台垫资余额（即时入金垫资）
- 银行通道状态（ACH/Wire/Plaid 各通道健康状态）

预警区:
- 待人工审核出金列表（超过 4 小时未处理标红）
- 今日 CTR 触发记录（>$10,000 USD / >HK$120,000）
- 对账异常告警
```

#### 1.3.2 出金审核队列

```
列表字段:
- 申请编号（参考号）、用户 ID、姓名、金额、币种
- 出金方式（ACH/Wire）、银行卡后 4 位
- 申请时间、等待时长
- 触发审核原因（金额超限 / 银行卡冷却期 / 新用户 / 风险评分）
- AML 筛查状态

审核操作:
- 点击进入详情：用户账户信息 + 出金历史 + AML 报告
- [批准] [拒绝] [上报合规主任]（对应 Rule 5 三层审批）
- 操作须填写处理说明，生成不可修改审计记录
- 大额出金（>$200,000）需两人审批（四眼原则）
```

#### 1.3.3 银行卡管理（管理员视角）

```
功能:
- 查看指定用户的所有银行卡（脱敏展示）
- 银行卡验证状态：Plaid 已验证 / 小额打款待确认 / 已验证 / 冷却期
- 手动解除/重置银行卡冷却期（需合规审批）
- 异常银行卡标记（如：银行账户名与 KYC 姓名不匹配的历史记录）
- 查看 Plaid 验证详情日志
```

#### 1.3.4 出入金记录查询

```
搜索条件:
- 用户 ID / 手机号 / 邮箱
- 交易参考号
- 日期范围
- 金额范围
- 状态（处理中/已完成/失败/退回）
- 币种 / 渠道

导出:
- CSV 导出（用于监管报告）
- 对账文件导出（与银行对账单格式匹配）
```

#### 1.3.5 对账管理

```
功能:
- 三方对账视图：内部账本 ↔ 银行流水 ↔ 托管账户
- 差异高亮：自动标红不匹配条目
- 手动标记已核对
- 对账状态：当日对账进度
- 历史对账报告查看（按日期）

自动告警规则（来自合规规则 Rule 6）:
- 差异 > $0.01：显示告警
- 差异 > $100：升级为紧急告警 + 操作限制提示
```

### 1.4 订单监控（Order Surveillance）

**背景**: v2/v3 定义了完整的订单状态机（待提交→已提交→待成交→部分成交→已成交/已撤销/已拒绝/已过期）和多种订单类型（市价/限价/止损/止损限价/追踪止损/MOO/MOC）。风险监控是监管合规的核心要求。

**所需页面**:

#### 1.4.1 实时订单流监控

```
核心功能（WebSocket 实时更新）:
- 实时订单流表格（虚拟滚动，支持 10 万+ 条）
- 每秒更新的订单状态变化
- 列：时间戳、用户、股票代码、方向(买/卖)、类型、数量、价格、状态、市场

高级筛选:
- 股票代码（支持批量输入）
- 订单状态
- 时间范围（精确到秒）
- 金额范围
- 用户 KYC 等级
- 标记/异常订单

告警规则:
- 单用户短时间内重复下单（频率告警）
- 异常大单（超出用户历史均值 3 倍）
- PDT 规则潜在违规（追踪当日交易次数）
- 价格偏离市价 5% 以上的限价单
```

#### 1.4.2 订单详情页

```
内容:
- 订单完整生命周期时间轴（每次状态变更 + 操作员操作）
- 执行明细（部分成交显示每笔成交记录）
- 关联的风控检查结果（通过/拦截/告警）
- 用户持仓上下文（下单时的持仓和可用资金）
- Best Execution 执行路径记录

操作:
- 手动撤销订单（需填写原因 + 审批）
- 标记为异常订单（触发 SAR 待审）
- 查看用户完整交易历史
```

#### 1.4.3 异常订单审查队列

```
自动入队条件:
- 系统风控规则命中
- 手动标记
- PDT 规则潜在违规
- 可疑交易模式（如洗售）

处理流程:
- 审查员查看 → 标记无异常 / 上报合规
- 合规员复审 → 关闭 / 提交 SAR
```

#### 1.4.4 PDT 规则追踪面板

```
功能:
- 显示所有账户当前 PDT 计数（当日日内交易次数）
- 接近 3 次限制的用户预警列表
- 账户净值 < $25,000 且 PDT 次数 ≥ 3 的告警
- 历史 PDT 违规记录查询
```

```
- 限制出金
- 提升/降低 KYC 等级
- 关闭账户

审批链:
- 操作员发起 → 主管审批（单人操作）
- 合规官发起 → 合规主任审批（合规操作）
- 所有批准/拒绝均记录到审计日志
```

### 1.6 合规报告中心（Compliance Center）

**背景**: 依据 fund-transfer-compliance.md 中的合规规则，以及 SEC Rule 17a-4、BSA、AMLO 等法规，平台必须提供完整的合规报告工具。

**所需页面**:

#### 1.6.1 AML 监控仪表板

```
实时监控区:
- 今日结构化交易告警数（Structuring Detection）
- 今日 CTR 触发数（自动/待人工确认）
- 待 JFIU 上报数（港股）
- 制裁名单命中记录（OFAC/SFC）

交易模式分析:
- 速进速出检测（rapid deposit → withdraw without trading）
- 高频小额拆分告警（多笔规避 CTR 阈值）
- 跨市场大额异常
```

#### 1.6.2 CTR 管理（Currency Transaction Report）

```
功能:
- 自动生成的 CTR 列表（>$10,000 USD 或 >HK$120,000）
- 状态：待确认 / 已提交 / 已归档
- CTR 详情：交易信息、客户信息、自动填充的 FinCEN 格式
- [确认并提交] [撤回并标记误报]
- 提交记录和时间戳（不可修改）
- 批量导出 FinCEN CTR 格式文件
```

#### 1.6.3 SAR 管理（Suspicious Activity Report）

```
SAR 工作流:
- 合规员标记可疑活动 → 填写 SAR 草稿
- SAR 草稿内容：可疑活动描述、金额、时间范围、涉及账户
- 合规主任审核 → 批准提交 / 退回修改
- 提交 FinCEN（US）或 JFIU（HK）
- 归档（7 年保留期，不可删除）

注意：SAR 存在后，相关用户账户不得被告知（Tipping-off prohibition）
后台需有权限控制，防止客服误操作通知用户
```

#### 1.6.4 审计日志查询（Audit Trail）

```
搜索维度:
- 时间范围
- 操作员 ID / 姓名
- 操作类型（KYC审核/出金审批/账户冻结/...）
- 涉及用户 ID
- 操作结果（成功/失败）

展示:
- 不可修改的时间序列记录
- 详细操作内容（before/after 对比）
- 关联的 IP 地址、设备信息
- 导出 CSV（SEC Rule 17a-4 合规格式）
```

#### 1.6.5 监管报告生成

```
报告类型:
- 月度 CTR 汇总报告
- AML 监控统计报告
- KYC 审核统计报告
- 用户开户 / 销户报告

导出格式:
- PDF（用于提交监管机构）
- CSV（用于内部分析）
- XML（FinCEN 规定的 BSA 报告格式）

报告权限:
- 仅合规官和主任可生成和导出监管报告
```

### 1.7 系统配置管理（Configuration Management）

**背景**: v3 中有多处业务规则数字（出金限额、KYC 等级、即时入金限额、银行卡冷却期等），这些必须可配置，不能硬编码。

**所需页面**:

#### 1.7.1 出入金规则配置

```
可配置项:
```

```
可配置项:

- GTC 订单最长有效期（当前：90 天）
- 各交易所交易时段配置（NYSE/NASDAQ/HKEX 时区和时段）
- 港股每手股数配置（按股票代码维护）
- 费率配置（佣金、交易所费等）
```

#### 1.7.3 功能开关（Feature Flags）

```
功能:
- 即时入金开关（Plaid 通道）
- 保证金账户开关（MVP 可能先关闭）
- 碎股交易开关
- 盘前盘后交易开关
- 港股竞价交易开关
- 游客模式开关

灰度发布:
- 按用户 ID 范围或 KYC 等级灰度放量
- 按地区/国家码灰度
```

### 2.2 Plaid 异常处理后台

v3 采用 Plaid 进行银行卡即时验证，但：

## 三、与移动端设计的一致性问题

### 3.1 订单状态枚举不一致

| 位置 | 状态定义 |
|------|---------|
| v2 设计文档（状态机）| 待提交 / 已提交 / 待成交 / 部分成交 / 已成交 / 已撤销 / 已拒绝 / 已过期 |

### 3.2 KYC 等级定义缺失

v3 出入金首页展示"KYC 等级: Tier 2"，但三版设计文档均未定义：
- Tier 1 / Tier 2 / Tier 3 各对应哪些已完成的 KYC 步骤
- 各 Tier 的出入金限额具体是多少
- 如何从 Tier 1 升级到 Tier 2（触发条件）

**影响**: 管理后台的出入金限额配置页面、用户详情页的限额显示都依赖这些定义。**PM 必须在产品规格文档中明确 KYC Tier 的完整定义。**

### 3.3 账户类型（现金/保证金）对管理后台的影响

v3 注册流程中有"账户类型选择（现金/保证金账户）"，但：
- 保证金账户的额外合规要求（Regulation T、保证金追缴规则）在管理后台如何体现？
- 保证金账户的审核流程是否与现金账户不同？
- MVP 是否仅支持现金账户（如果是，管理后台可以暂不支持保证金相关功能）

**影响**: 如果 MVP 同时支持两种账户类型，管理后台的复杂度将显著增加。建议 PM 明确 MVP 范围。

### 3.4 风险评分（Risk Score）的定义缺失

出金合规规则中多次引用"风险评分（Risk Score: LOW/MEDIUM/HIGH）"，但：
- 风险评分如何计算（哪些因素、权重）
- 风险评分由哪个系统生成（AML 系统？内部规则引擎？）
- 管理后台是否需要支持手动调整风险评分（需审批）

**影响**: 管理后台的出金审核页面、用户详情页均需显示和处理风险评分。

### 3.5 港股交易数据模型的缺失

UX 设计师在 design-review-for-pm.md 中已指出港股设计几乎为零，这对管理后台同样有影响：
- 港股订单的货币单位是 HKD 还是 USD（对账影响）
- 港股的"手"（Board Lot）单位在订单监控中如何展示
- HKEX 的竞价时段（9:00-9:30 HKT）订单如何在管理后台处理

### 4.2 关键技术决策

#### 4.2.1 实时数据架构

管理后台需要两类实时数据：

**高频实时**（订单流、系统健康状态）:
- 使用 WebSocket 持久连接
- Zustand 维护内存中的实时数据状态
- ProTable 配合虚拟滚动（`rc-virtual-list`）渲染大数据集
- 批量更新策略：每 500ms 批处理一次 WebSocket 消息，避免频繁 React re-render

**中频实时**（出入金待审核计数、KYC 队列长度）:
- TanStack Query + 轮询（10 秒间隔）
- 或 WebSocket + 事件推送

#### 4.2.2 证件图片查看器

KYC 审核的核心 UI 组件，需要定制开发：

```
技术方案:
- 基于 react-image-crop 或 react-zoom-pan-pinch
- 图片加载：懒加载 + 渐进式渲染（证件图可能 > 5MB）
- 安全要求：
  - 图片 URL 使用有时效的预签名 URL（S3 Presigned URL，15 分钟有效期）
  - 禁止图片右键下载（CSS user-select: none + pointer-events）
  - 操作日志：记录每次证件图查看（合规要求）
- 对比视图：证件照 vs 人脸识别截帧并排展示
```

#### 4.2.3 审批工作流

出金审核、账户冻结等操作需要多层审批，建议使用状态机模式：

```typescript
// 出金审批状态机
type WithdrawalApprovalState =
  | 'PENDING_REVIEW'        // 待一级审核
  | 'PENDING_COMPLIANCE'    // 待合规主任
  | 'APPROVED'              // 已批准
  | 'REJECTED'              // 已拒绝
  | 'ESCALATED';            // 已升级

// 每次状态转换生成不可变审计记录
interface ApprovalAction {
  fromState: WithdrawalApprovalState;
  toState: WithdrawalApprovalState;
  operatorId: string;
  reason: string;
  timestamp: string; // ISO 8601 UTC
  ipAddress: string;
}
```

#### 4.2.4 财务数字展示规范

与移动端相同，管理后台**严禁**使用 `number` 类型进行财务计算：

```typescript
// 正确：使用 big.js 处理财务数值
import Big from 'big.js';

const formatUSD = (amount: string): string => {
  return new Intl.NumberFormat('en-US', {
    style: 'currency',
    currency: 'USD',
    minimumFractionDigits: 2,
    maximumFractionDigits: 2,
  }).format(new Big(amount).toNumber());
};

// 错误：绝对禁止
const total = parseFloat(amount) + fee; // FORBIDDEN
```

#### 4.2.5 SAR 保密性的前端处理

SAR（可疑活动报告）有特殊的"禁止告知"（Tipping-off）法律要求，前端需要：

```
技术控制:
- SAR 相关 UI 仅在合规官/主任角色下可见
- 涉及 SAR 的用户账户页面，需要隐藏 SAR 入口（即使客服进入同一用户详情页）
- 所有 SAR 相关 API 调用单独审计日志
- SAR 草稿本地不缓存（不写入 localStorage/sessionStorage）
```

### 6.3 管理员操作审计要求

所有改变系统状态的操作必须记录：

```typescript
interface AdminAuditLog {
  id: string;
  timestamp: string;       // ISO 8601 UTC，不可修改
  operatorId: string;      // 操作人员 ID
  operatorRole: AdminRole; // 操作时的角色
  action: string;          // 操作类型（枚举）
  resourceType: string;    // 操作资源类型
  resourceId: string;      // 操作资源 ID（用户 ID / 订单 ID 等）
  before: unknown;         // 操作前状态（JSON）
  after: unknown;          // 操作后状态（JSON）
  reason: string;          // 操作原因（必填）
  ipAddress: string;       // 操作 IP
  userAgent: string;       // 浏览器信息
  correlationId: string;   // 关联的请求 ID
}
```

**审计日志不得修改、不得删除**，存储于 append-only 存储（或 Elasticsearch 只读索引）。

### 6.4 会话安全

```
要求:
- Admin Panel 独立的认证系统（不与移动端 JWT 共用）
- Admin 用户必须启用 MFA（TOTP 或硬件密钥）
- 会话超时：30 分钟无操作自动登出（可配置）
- 同一账户不允许多设备并发登录
- 所有 API 请求附带操作者 Session ID
- 敏感操作（证件图查看、SAR 提交）需要 Step-up Authentication
```

## 七、合规需求

### 7.1 数据保留与显示

| 数据类型 | 保留要求 | 管理后台实现 |
|---------|---------|------------|
| 出入金记录 | 7 年 | 归档数据查询入口（冷数据，加载稍慢，有提示）|
| AML 筛查记录 | 7 年 | 同上 |
| CTR/SAR 归档 | 5 年 | 只读查看，不可删除 |
| KYC 审核记录 | 6 年（账户关闭后）| 软删除，后台可查 |
| 审计日志 | 7 年 | Elasticsearch 只读索引 |
| Admin 操作日志 | 7 年 | 同上 |

### 7.2 数据脱敏规则

管理后台展示敏感数据时必须遵循以下脱敏规则：

```typescript
// 脱敏工具函数（所有展示层调用，绝不在日志中记录原始值）
const mask = {
  ssn: (v: string) => `***-**-${v.slice(-4)}`,
  bankAccount: (v: string) => `****${v.slice(-4)}`,
  hkid: (v: string) => `${v[0]}****(${v.slice(-1)})`,
  email: (v: string) => {
    const [local, domain] = v.split('@');
    return `${local[0]}***${local.slice(-1)}@${domain}`;
  },
  phone: (v: string) => `${v.slice(0, 3)}****${v.slice(-4)}`,
};

// 合规官解密权限：通过专门的 API 端点，调用行为记录审计日志
```

### 7.3 WORM 合规（Write Once Read Many）

监管报告和审计日志存储必须满足 SEC Rule 17a-4 的 WORM 要求：
- Admin Panel 的"导出"功能在后端生成不可变副本
- 导出记录本身也须记录到审计日志（谁、何时、导出了什么）
- 长期归档数据（>2 年）通过独立的只读入口访问，不允许在线编辑

| **第 4 阶段** | 订单监控：实时 WebSocket 表格、告警、详情页、PDT 追踪 | 3 周 |
| **第 5 阶段** | 用户管理 + 合规中心：用户详情、账户操作、CTR、审计日志 | 3 周 |
| **第 6 阶段** | 联调测试、合规验收、性能调优、安全审计 | 2 周 |

**MVP 总工期估算: 17 周（约 4.5 个月）**

### 9.3 风险提示

- **API 依赖风险**: Admin Panel 强依赖后端 API，若后端 OpenAPI 规范延迟提供，前端开发将受阻。建议第 1 周就确定 API 契约（Contract-first）。
- **合规需求变更风险**: 监管要求可能在开发过程中变化（如 ASPIRe Roadmap 更新），建议合规模块预留 20% 缓冲。
- **实时数据复杂度风险**: 订单监控的实时数据量可能超出预估，如果日均订单超过 10 万笔，需要评估 WebSocket 连接池和前端虚拟滚动方案。
