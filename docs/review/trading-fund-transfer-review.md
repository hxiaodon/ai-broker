> **评审日期**: 2026-03-11
> **评审对象**: mobile-app-design v1/v2/v3 设计文档、UX 设计师评审反馈

# Part 1: 交易引擎视角 (Trading Engine Review)

---

## 1. 订单类型与状态机

### 1.1 订单类型覆盖度评估

v3 设计文档中列出的订单类型：

| 订单类型 | 设计文档提及 | 交易引擎必须支持 | 评估 |
|---------|------------|----------------|------|
| Market (市价单) | 有 | 是 | 合格 |
| Limit (限价单) | 有 | 是 | 合格 |
| Stop (止损单) | 有 | 是 | 合格 |
| Stop Limit (止损限价单) | 有 | 是 | 合格 |
| Trailing Stop (追踪止损) | 有 (仅美股) | 是 | 合格，但需明确仅 US market |
| MOO (市价开盘单) | 有 (在 v3 高级展开) | 是 | 合格 |
| MOC (市价收盘单) | 有 (在 v3 高级展开) | 是 | 合格 |
| AON (全部成交否则取消) | **未提及** | 是 (US only) | **缺失** |
| IOC (即时成交剩余取消) | 有 | 是 | 合格 |
| FOK (全部成交或全部取消) | 有 | 是 | 合格 |
| GTC (持续有效) | 有 (最长90天) | 是 | 合格，需明确服务端 GTC 到期自动撤单逻辑 |
| DAY (当日有效) | 有 | 是 | 合格 |
| GTD (指定日期有效) | **未提及** | 建议支持 | **缺失** — 部分竞品支持，建议 MVP 后增加 |
| LOO (限价开盘单) | **未提及** | 建议支持 (HK) | **缺失** — 港股竞价时段需要 |
| LOC (限价收盘单) | **未提及** | 建议支持 (HK) | **缺失** — 港股收市竞价时段需要 |

**关键问题**:

1. **AON 缺失**: 设计中 FOK 和 IOC 都有，但 AON 遗漏。AON 允许订单留在 book 上直到能全部成交，区别于 FOK 的立即性。需要确认产品是否需要。
2. **港股竞价限价盘 (LOO/LOC)**: v3 提到了收市竞价时段 (16:00-16:10 HKT)，但没有对应的 LOC 订单类型。港股开市前时段 (09:00-09:30) 同样需要竞价限价盘。设计文档中的 "提前设置限价单，13:00 后提交" 描述不精确 — 在交易引擎中这应该映射为预约单（scheduled order），不是标准限价单。
3. **GTC 到期处理**: 设计文档提到"最长90天"，但未设计到期提醒通知。交易引擎需要 GTC 订单的过期清理 job，以及到期前 N 天的用户通知。
4. **Trailing Stop 仅限美股**: 设计文档在买入页提到了追踪止损，但未明确标注仅限美股。HKEX 不支持 trailing stop 订单类型，如果用户在港股页面看到此选项但无法使用，会造成困惑。

### 1.2 订单状态机完整性

v3 订单详情页时间轴隐含的状态流转：

```
订单创建 → 风控通过 → 发送交易所 → 交易所确认 → 部分成交 → 等待剩余成交
```

对照交易引擎实际需要的状态机：

| 状态 | 设计文档体现 | 评估 |
|------|------------|------|
| CREATED (已创建) | 有 — "订单创建" | 合格 |
| VALIDATED (已验证) | **未区分** — 合并在风控通过中 | **缺失** — 需区分参数校验和风控审批 |
| APPROVED (风控通过) | 有 — "风控通过" | 合格 |
| REJECTED (被拒绝) | 有 — 下单失败处理表 | 合格 |
| PENDING (已发送待确认) | 有 — "发送交易所" | 合格 |
| OPEN (交易所已确认) | 有 — "交易所确认" | 合格 |
| PARTIALLY_FILLED (部分成交) | 有 — "部分成交" | 合格 |
| FILLED (全部成交) | 有 (隐含) | 合格 |
| CANCEL_REQUESTED (撤单请求中) | **未明确** | **缺失** — 撤单有中间态 |
| CANCELLED (已撤单) | 有 — 撤单确认弹窗 | 合格 |
| EXPIRED (已过期) | **未提及** | **缺失** — DAY 单收盘自动过期 |
| REPLACE_REQUESTED (改单请求中) | **未明确** | **缺失** — 改单有中间态 |
| REPLACED (已改单/被替代) | 有 (隐含于改单流程) | 需明确旧单标记为 REPLACED |

**关键问题**:

1. **CANCEL_REQUESTED 中间态缺失**: 设计文档的撤单流程是"确认撤单 → 完成"。但实际上，撤单请求发送到交易所后，存在一个等待交易所确认的窗口期。在此期间，订单可能继续成交。用户需要看到"撤单请求中"状态，以及撤单可能失败（因为在撤单确认前已全部成交）的情况。v3 撤单弹窗提到了这个场景（"如交易所在撤单前完成了成交，则撤单失败"），但没有对应的 UI 状态设计。

2. **REPLACE_REQUESTED 中间态缺失**: 改单同理。改单提交后也需要等待交易所确认。如果交易所拒绝改单请求（如价格不合规），用户看到什么？

3. **EXPIRED 状态缺失**: DAY 单在市场收盘后自动过期。GTC 单到期后也过期。设计文档没有 EXPIRED 状态的 UI 展示和通知推送设计。

4. **部分成交后改单/撤单的状态流转**: v3 改单页面做得不错，明确了"已成交的50股不受影响"。但在状态机层面，部分成交后改单，实际上是 cancel 原单剩余部分 + 创建新单。旧单应标记为 REPLACED，新单从 CREATED 开始走。两个单之间需要 `replaced_by_order_id` / `replaces_order_id` 关联。

### 1.3 状态转换合法性矩阵

交易引擎必须强制执行以下状态转换规则（invalid transition 立即 reject）:

| 当前状态 | 允许转换到 |
|---------|----------|
| CREATED | VALIDATED, REJECTED |
| VALIDATED | APPROVED, REJECTED |
| APPROVED | PENDING, REJECTED (risk re-check failure) |
| PENDING | OPEN, REJECTED (exchange reject) |
| OPEN | PARTIALLY_FILLED, FILLED, CANCEL_REQUESTED, REPLACE_REQUESTED |
| PARTIALLY_FILLED | PARTIALLY_FILLED (再次部分成交), FILLED, CANCEL_REQUESTED, REPLACE_REQUESTED |
| CANCEL_REQUESTED | CANCELLED, FILLED (race condition: 撤单前已全部成交), PARTIALLY_FILLED (race: 撤单前又成交了一部分) |
| REPLACE_REQUESTED | REPLACED, FILLED (race), PARTIALLY_FILLED (race), OPEN (replace rejected, 回到原状态) |
| FILLED | (终态，不可转换) |
| CANCELLED | (终态，不可转换) |
| EXPIRED | (终态，不可转换) |
| REJECTED | (终态，不可转换) |
| REPLACED | (终态，不可转换) |

**设计文档需要补充**: 终态不可操作的 UI 禁用规则 — FILLED/CANCELLED/EXPIRED/REJECTED 状态下，改单和撤单按钮应 disable 或隐藏。

## 2. 交易流程合规性

### 2.1 PDT (Pattern Day Trader) 规则

**设计文档现状**: v3 下单失败处理表提到 "PDT限制：5日内已有3次日内交易"，交易设置有"显示PDT警告"开关。

**合规要求 vs 设计差距**:

| 要求 | 设计是否满足 | 差距分析 |
|------|------------|---------|
| 跟踪滚动5个工作日内的日内交易次数 | 部分 — 提到了次数限制 | 需明确：日内交易 = 同一股票当天买入又卖出（或先卖后买）|
| 账户权益 < $25,000 时，5个工作日内不超过3次日内交易 | 部分 | 设计说"5日内已有3次"，应该是"即将达到4次"时阻断 |
| **必须阻断，不能仅警告** | **不满足** | 交易设置中"显示PDT警告"是一个开关，暗示可以关闭。**这是合规红线** — PDT 限制不是可选警告，是 FINRA 强制要求，必须在交易引擎层面硬拦截 |
| 账户被标记为 PDT 后的限制 | 未设计 | 被标记为 PDT 后的解除流程、限制说明页面缺失 |
| 仅适用于 margin 账户 | 未明确 | 设计文档未区分 cash 账户不受 PDT 限制（但 cash 账户有 free-riding 限制）|
| Day Trade Buying Power (DTBP) | 未提及 | PDT 账户有 4x DTBP，普通 margin 有 2x。设计未体现 |

**严重问题**: "显示PDT警告"设置开关必须删除。PDT 限制是法规要求，交易引擎在下单前必须检查，不通过则直接 reject，不存在"关闭警告"的选项。交易设置页面可以保留 PDT 规则说明入口，但不能有关闭拦截的开关。

**建议的 PDT 实现方案**:

1. 交易引擎维护每个 margin 账户滚动 5 个工作日的日内交易计数
2. 下单时，如果该笔交易会构成日内交易（当天已有反向持仓），检查 PDT 计数
3. 如果计数将达到 4 次且账户权益 < $25,000，硬拒绝并返回明确原因
4. UI 层面：在第 3 次日内交易成交后，推送提醒 "您已有3次日内交易，再增加1次将触发PDT限制"
5. 账户被标记为 PDT 后，限制交易直到权益恢复 $25,000 以上

### 2.2 Best Execution 义务

**设计文档现状**: v3 订单确认弹窗包含 "Best Execution" 信息区域，显示执行场所和 NBBO。

**正面评价**: 设计中包含了 NBBO 价格展示、执行场所标注和预估滑点，这是很好的起点。

**缺失项**:

1. **订单执行质量报告 (SEC Rule 605/606)**: 设计未涉及执行质量统计页面。Rule 606 要求披露订单路由信息，包括 PFOF (Payment for Order Flow) 信息。需要增加一个"执行质量报告"页面入口。
2. **PFOF 披露**: 如果平台接受 PFOF（零佣金模式通常依赖 PFOF），必须在下单前披露。设计中佣金显示 $0.00，但未披露可能的 PFOF。
3. **市价单滑点保护**: 市价单在流动性差的股票上可能产生极大滑点。设计未体现市价单的价格保护机制（如 collar — 拒绝偏离 NBBO 超过 X% 的成交）。
4. **Reg NMS Order Protection Rule**: 交易引擎必须确保不以劣于 NBBO 的价格成交。这对 SOR 有直接影响，设计文档完全未提及。

### 2.3 Pre-trade Risk Checks 在 UI 中的体现

v3 下单失败处理表列出了部分风控拒绝场景，但设计文档没有系统性地描述 pre-trade 风控管线。以下是交易引擎需要执行的风控检查及其在 UI 中的映射：

| 风控检查 | 设计是否有对应 UI | 差距 |
|---------|----------------|------|
| 1. 账户状态检查 (active, not frozen) | 无 | 需增加账户冻结态下所有交易入口的禁用设计 |
| 2. 标的状态检查 (停牌/熔断) | 有 — 停牌提示页 | 合格，但缺少"熔断"(circuit breaker) 状态 |
| 3. 订单有效性校验 (价格/数量) | 部分 — 港股 tick size 有提示 | 美股 tick size ($0.01) 未强调；最小/最大数量限制未设计 |
| 4. 购买力检查 | 有 — "余额不足"提示 | 合格，但购买力计算细节未在设计中体现 |
| 5. 持仓限制检查 | 无 | 缺少集中度限制 (单只股票不超过组合 X%) 的 UI 提示 |
| 6. 频率限制 (防操纵) | 无 | 高频下单拦截无 UI 设计 |
| 7. PDT 检查 | 有 — 但实现有误 (见 2.1) | 需修正 |
| 8. 保证金检查 | 部分 — 显示可用保证金 | Reg T 50% 初始保证金、25% 维持保证金的具体计算未设计 |
| 9. 价格偏离检查 | 有 — 交易设置中"偏离市价>5%" | 合格，但应为强制检查，不仅仅是用户设置 |
| 10. 市价单金额上限 | 无 | 市价单应有金额上限（如 $100,000），超过强制使用限价单 |

## 3. 美股 vs 港股交易规则差异

### 3.1 交易时段

**设计文档现状**: v3 有盘前盘后 UI 和港股午休提示，覆盖度不错。

**详细差异及引擎需要处理的逻辑**:

| 时段 | 美股 (ET) | 港股 (HKT) | 设计覆盖 | 引擎注意事项 |
|------|----------|-----------|---------|------------|
| 盘前交易 | 04:00-09:30 | 无 | 有 | 仅限限价单，正确 |
| 开市前时段 | 无 | 09:00-09:30 | **缺失** | 港股开市前竞价，需支持竞价限价盘 |
| 上午连续交易 | 09:30-16:00 | 09:30-12:00 | 有 | -- |
| 午休 | 无 | 12:00-13:00 | 有 | 午休期间订单行为需定义：已挂限价单是否保留？ |
| 下午连续交易 | (同上) | 13:00-16:00 | 有 | -- |
| 收市竞价 | 无 (MOC 由交易所处理) | 16:00-16:10 | 有 | 需支持竞价限价盘 (at-auction limit order) |
| 盘后交易 | 16:00-20:00 | 无 | 有 | 仅限限价单，正确 |

**关键缺失**:

1. **港股开市前时段 (09:00-09:30)**: 设计完全没有涉及。这是港股的竞价时段，交易引擎需要支持竞价盘 (at-auction order) 和竞价限价盘 (at-auction limit order)。UI 需要类似"收市竞价时段"的提示弹窗。

2. **港股午休期间已挂单行为**: 午休提示说"午休期间不接受新订单"，但未说明已有的挂单如何处理。交易引擎中，上午未成交的限价单在午休后是否继续有效需要明确（HKEX 规则：持续交易时段的挂单在午休后自动延续）。

3. **美股半日交易**: 美国节假日前一天常有半日交易 (13:00 ET 收盘)。设计未涉及。交易引擎需要一个交易日历服务。

4. **时区转换显示**: v3 设计在市场休市页显示了"(北京时间 22:30)"，很好。但需要系统化 — 所有交易时间展示都应支持用户本地时区转换。

### 3.2 价格与数量规则

| 规则 | 美股 | 港股 | 设计覆盖 |
|------|------|------|---------|
| 最小报价单位 (Tick Size) | $0.01 (统一) | 按价格区间浮动 | 港股有，美股未提及 |
| 最小交易单位 | 1 股 (fractional shares 另议) | 每手 100/200/400/500/1000/2000 股不等 | 港股有 (以100股为例) |
| 碎股交易 | 部分券商支持 | 有碎股市场 (但流动性差) | **均未设计** |
| 涨跌幅限制 | 无 (但有 LULD 熔断) | 无 (但 VCM 冷静期) | 无 — LULD 和 VCM 都未设计 |
| 最大报价偏离 | LULD bands | 价格区间 +-5% (收市竞价) | 收市竞价有提到 +-5% |

**关键问题**:

1. **港股 Tick Size 表**: v3 中仅展示了"价格区间 200-500 的 Tick Size: HK$0.20"。但港股 tick size 有 14 个价格区间档位。交易引擎必须内置完整的 HKEX tick size 表，并在下单时强制校验价格是否符合 tick size。UI 的步进器 (+/-) 按钮必须动态调整步长。

2. **港股 Board Lot 多样性**: v3 以腾讯 (100股/手) 为例，但港股不同股票的手数不同（如汇丰 400股/手）。交易引擎需要从 HKEX 证券主档获取每只股票的 board lot size，并在 UI 动态展示。设计中硬编码"须为 100 的整数倍"是错误的。

3. **LULD (Limit Up/Limit Down)**: 美股 SEC 的价格波动限制机制。当股价在 5 分钟内偏离参考价超过一定百分比时，进入 Limit State 甚至 Trading Pause。交易引擎必须处理 LULD 状态下的订单行为（限价单价格不能超出 LULD band），但设计文档完全未涉及。

4. **VCM (Volatility Control Mechanism)**: 港股的波动调节机制。当 AMS 价格偏离参考价超过 10% 时触发 5 分钟冷静期，期间仅接受限价盘。交易引擎需要处理 VCM 触发后的订单限制。

### 3.3 结算周期

| 项目 | 美股 | 港股 | 设计覆盖 |
|------|------|------|---------|
| 结算周期 | T+1 (自2024年5月) | T+2 | v3 卖出页有"未结算: 20股 (03-12 可卖)" |
| 未结算股票能否卖出 | 可以 (但有 free-riding 限制) | 可以 (但卖出款 T+2 才到账) | 部分覆盖 — "可卖(已结算)"区分做得好 |
| Free-riding 违规 | Cash 账户：用未结算资金买入并卖出 → 90天限制 | 类似规则 | **完全未设计** |
| Good Faith Violation | 卖出后用未结算资金再次买入并卖出 | 不适用 | **完全未设计** |

**关键问题**:

1. **Free-riding 和 Good Faith Violation**: 这是 cash 账户的重要合规约束。交易引擎必须追踪每一笔买入资金的来源（settled vs unsettled），并在卖出时检查是否构成违规。设计文档没有 cash 账户特有限制的 UI 说明。

2. **结算日计算**: 需要考虑美股和港股的不同假日日历。T+1 和 T+2 中的"T"是 trade date，"+1/+2" 是工作日（排除交易所假日）。交易引擎需要维护 NYSE/NASDAQ 和 HKEX 的假日日历。

### 3.4 卖空规则

**设计文档现状**: 完全未涉及卖空 (short selling)。

| 规则 | 美股 (Reg SHO) | 港股 | 引擎需求 |
|------|---------------|------|---------|
| Locate 要求 | 卖空前必须确认可借入股票 | 类似 — 需预先安排 | 必须实现 |
| 裸卖空 | 禁止 | 禁止 | 必须在风控中拦截 |
| Uptick Rule (SSR) | SEC Rule 201: 股价跌幅 >10% 时触发 SSR | 无 (但有卖空报告义务) | 引擎需实时监控 SSR 状态 |
| 报告义务 | FINRA short interest reporting | SFC 淡仓申报 | 后台合规系统 |

**如果 MVP 不支持卖空**: 交易引擎必须明确拒绝卖出数量 > 持仓数量的订单。这看似简单，但在部分成交场景中需要特别小心 — 比如持仓 100 股，挂了两个各卖 100 股的限价单，如果第一个全部成交后第二个也开始成交，就构成了卖空。

**建议**: 设计文档需要明确："MVP 不支持卖空"，并在设计中体现：(1) 卖出数量不能超过可卖数量；(2) 有未成交卖单时，可卖数量需扣除挂单数量。

## 4. 订单修改与撤单

### 4.1 改单机制

**设计文档现状**: v3 有改单页面 (3.6)，正确地指出"改单将取消原订单并提交新订单"。

**技术实现要点**:

1. **Cancel/Replace (FIX MsgType=G)**: 在 FIX 协议中，改单通过 OrderCancelReplaceRequest (MsgType=G) 实现。交易所要么 accept (返回新的 execution report with ExecType=5 Replaced)，要么 reject。这不是两步操作，而是原子性的 — 要么整体成功，要么整体失败，不会出现取消成功但新单失败的中间态。

2. **可修改范围的约束**:
   - 价格: 可修改 (限价单、止损价)
   - 数量: 只能减少，不能增加（增加数量在大多数交易所会被拒绝或被视为新单）
   - 订单类型: 不可修改（如果要从限价改市价，需要 cancel + new）
   - 有效期: 通常不可修改
   - Side: 绝对不可修改

3. **设计问题**: v3 改单页面允许修改数量范围是"1-50股"（原剩余量），但未说明不能超过原始订单数量。如果原单 100 股，已成交 50 股，剩余 50 股 — 改单可以将剩余从 50 改为 30（减少），但不能改为 70（增加超过原剩余）。设计需要更精确地表达这个约束。

4. **竞争条件**: 用户点击改单时，如果恰好有新的成交发生，剩余量变化了，改单页面展示的信息会过期。需要设计：(a) 实时更新改单页面的已成交/剩余信息；(b) 改单请求发送时做乐观锁检查；(c) 改单被拒绝后的友好提示。

### 4.2 撤单机制

**设计文档现状**: v3 撤单确认弹窗 (3.7) 设计合理，明确了部分成交不受影响。

**补充要点**:

1. **撤单竞争**: 用户发起撤单到交易所确认之间有延迟。此期间如果订单成交了，撤单会失败。UI 需要处理这种情况 — "撤单失败，订单已全部成交"。
2. **港股收市竞价时段不可撤单**: v3 已经提到"不可取消已提交的订单"，正确。交易引擎需要在收市竞价时段禁用撤单功能。
3. **批量撤单**: 设计未提及。如果用户有多个挂单，是否需要"一键撤销所有订单"功能？竞品普遍支持。
4. **收盘自动撤单**: DAY 单在收盘后由交易所自动撤销。交易引擎会收到交易所发来的 Cancelled execution report。需要推送通知用户"您的 X 个 DAY 单已在收盘后自动撤销"。

## 5. Smart Order Router (SOR)

**设计文档现状**: v3 订单确认弹窗中有"执行场所: NYSE (最优报价)"，但这只是展示层面。设计文档完全没有涉及 SOR 的逻辑。

### 5.1 美股 SOR 需求

美股有多个执行场所：NYSE, NASDAQ, ARCA, BATS, IEX, 以及 dark pools 和 wholesalers (Citadel, Virtu 等)。Reg NMS 要求 best execution，即不能以劣于 NBBO 的价格成交。

**交易引擎需要实现**:

1. **NBBO 获取**: 实时聚合各交易所的 BBO (Best Bid/Offer)，计算 NBBO
2. **路由决策因素**:
   - 价格优先 (Price improvement 机会)
   - 执行概率 (流动性深度)
   - 延迟 (各场所的连接延迟)
   - 成本 (maker-taker 费用模型，rebate)
   - 订单大小 (大单可能需要拆分或使用 dark pool)
3. **PFOF 模式**: 如果平台采用零佣金模式将散户订单发送给 wholesaler (如 Citadel Securities)，必须:
   - 披露 PFOF 安排 (SEC Rule 606 报告)
   - 证明 wholesaler 提供的价格不劣于 NBBO (price improvement)
   - v3 风险披露中的"Best Execution 披露"需要包含 PFOF 信息

4. **IEX 优先**: 如果平台定位为"对散户友好"，可以考虑优先路由到 IEX (Investors Exchange)，其 speed bump 机制对散户有利。

### 5.2 港股 SOR 需求

港股交易集中在 HKEX，不存在多交易所竞争。但需要考虑：

1. **暗盘 (Dark Pool)**: 港股有场外暗盘交易。如果平台提供暗盘交易，需要 SOR 支持。
2. **市场微结构差异**: HKEX 使用 price-time priority 且没有 maker-taker 费用模型。

### 5.3 设计文档需要补充

1. 下单确认中的"执行场所"应该是 SOR 的输出，而不是固定写 NYSE
2. 订单详情 / 成交明细中应该显示实际执行场所（每笔成交可能在不同场所）
3. 增加 Rule 606 报告入口（可以在账户设置或法律文件中）
4. 限价单的路由策略可能与市价单不同，设计中未区分

## 6. FIX Protocol 对接

**设计文档现状**: 未涉及。这是纯后端话题，但有若干点需要 UI 配合。

### 6.1 对 UI 设计的影响

1. **消息序号恢复 (Sequence Number Recovery)**: FIX 会话断开重连后需要 gap fill。在此期间（可能几秒到几十秒），订单提交和状态更新会有延迟。UI 需要设计连接恢复中的状态 — 如 "交易通道恢复中，请稍候"。

2. **Exchange Reject**: 交易所通过 FIX Execution Report (ExecType=8 Rejected) 拒绝订单。拒绝原因需要映射为用户可理解的文案。v3 的"交易所拒绝：{原因}"需要一个完整的原因码映射表。

3. **Business Reject (MsgType=j)**: 消息级别的拒绝（如字段缺失、无效值），需要与订单级别的 reject 区分处理。

4. **Execution Report 延迟到达**: 网络抖动可能导致 execution report 延迟到达。用户下单后可能出现较长时间处于 PENDING 状态。UI 需要设计超时处理 — 如果 30 秒没有收到交易所确认，提示用户 "订单处理时间较长，请在订单列表查看最新状态"。

### 6.2 FIX Session 健康状态

交易引擎需要暴露 FIX session 健康状态给前端：

| 状态 | UI 表现 |
|------|---------|
| Connected & Healthy | 正常（绿色指示灯或无显示） |
| Reconnecting | 顶部黄色 banner "交易通道重连中" |
| Disconnected | 顶部红色 banner "交易通道中断"，禁用下单 |
| Market Closed (session not established) | 正常显示 "市场已休市" |

**设计文档缺失**: 缺少交易通道连接状态的 UI 指示器设计。

## 7. Pre-trade Risk Controls (风控管线详细分析)

### 7.1 购买力计算

v3 设计中"可用资金"和"可用保证金"的展示是正确的方向，但计算细节对交易引擎至关重要：

**Cash 账户购买力**:
```
BuyingPower = SettledCash + UnsettledProceeds - PendingBuyOrders - PendingWithdrawals
```

**Margin 账户购买力**:
```
BuyingPower = max(0, (MarginEquity - MaintenanceMargin) / InitialMarginRate - PendingBuyOrders)
```
其中 InitialMarginRate = 50% (Reg T)

**设计问题**:

1. v3 买入页显示"可用资金: $10,000.00 USD"和"可用保证金: $20,000.00" — 这里的"可用保证金"是否是购买力？如果是 margin 账户，$10,000 可用资金 + 2x 杠杆 = $20,000 购买力，计算是合理的。但设计没有明确展示计算方式。
2. 佣金和预估费用是否从购买力中扣除？应该扣除。"最大可买 57 股"的计算必须包含佣金预估。
3. 多币种购买力：如果用户 USD 不足，能否自动换汇使用 HKD 余额购买美股？设计未涉及。

### 7.2 保证金追缴 (Margin Call)

**设计完全缺失**。

保证金账户必须维持 25% 维持保证金率（部分高波动标的更高）。当持仓市值下跌导致保证金率不足时：

1. **Margin Warning**: 保证金率接近维持保证金时推送警告
2. **Margin Call**: 保证金率跌破维持保证金时发出追缴通知
3. **强制平仓 (Forced Liquidation)**: 用户未在规定时间内补足保证金，系统自动卖出持仓

**需要增加的 UI 设计**:
- 保证金率实时展示（在持仓页面或账户概览）
- Margin Warning 推送通知
- Margin Call 页面（显示缺口金额、补缴期限、可能被强平的持仓）
- 强制平仓通知

### 7.3 风控拦截的用户体验

v3 下单失败处理表是好的起点，但需要系统化：

**每种风控拒绝都需要**:
1. 明确的拒绝原因（不能是笼统的"风控拒绝"）
2. 可操作的解决建议（入金、调整数量、联系客服）
3. 不泄露内部风控参数（如不要说"您的订单频率超过了每分钟 10 次的限制"）

## 9. 持仓与盈亏计算

### 9.1 设计文档现状

v3 卖出页展示了持仓信息：持有数量、均价、市值、浮盈、可卖(已结算)、未结算。成交记录页有成交汇总。

### 9.2 P&L 计算方法

**交易引擎必须同时支持两种计算方法**:

1. **Average Cost (平均成本法)**: 用于实时展示和保证金计算
   ```
   AvgCost = TotalCostBasis / TotalQuantity
   UnrealizedPnL = (MarketPrice - AvgCost) * Quantity
   ```

2. **FIFO (先进先出法)**: 用于税务报告 (IRS requirement)
   ```
   卖出时，先消耗最早买入的 lot
   每个 lot 有独立的成本基础和持有期（区分长期/短期资本利得）
   ```

**设计问题**:

1. v3 展示的"均价"是 average cost，适用于日常展示。但需要在某个地方（如持仓详情或报税页面）提供 FIFO 视角的 lot-level 信息。
2. **Wash Sale 规则**: 卖出亏损后 30 天内买回同一股票，亏损不能用于抵税，需要加回到新买入的成本基础。交易引擎需要追踪 wash sale，并在报税相关页面标注。设计文档未涉及。
3. **多币种 P&L**: 港股持仓以 HKD 计价，但用户可能希望看到 USD 等值的 P&L。汇率波动也是盈亏的一部分。设计未涉及多币种 P&L 合并展示。

### 9.3 结算感知的持仓管理

v3 卖出页的"可卖(已结算): 80 股"设计做得很好。交易引擎需要：

1. 每笔买入交易记录结算日期 (settlement date)
2. 每日 batch job 更新结算状态
3. 可卖数量 = 已结算数量 - 已挂卖单数量（v3 未扣除已挂卖单）

**问题**: v3 设计中"可卖(已结算): 80 股"旁边的快捷按钮"[全部可卖: 80]"没有考虑已有的挂卖单。如果用户已经有一个挂单卖 30 股 AAPL，那实际可卖应该是 80-30=50 股。

## 12. 入金方式评估

### 12.1 USD 入金

| 方式 | v3 设计覆盖 | 手续费 | 到账时间 | 评估 |

| Check Deposit | 未提及 | -- | -- | 可暂不支持，但竞品有 |

**ACH 技术细节**:
- ACH 入金需要通过 NACHA 网络，支持 same-day ACH（额外费用）和 regular ACH
- ACH 有退回风险 (ACH Return) — 常见原因：余额不足 (R01)、账户关闭 (R02)、未授权 (R10)
- ACH Return 可在 T+2 到 T+60 天内发生（Unauthorized 类型可达 60 天）
- **设计缺失**: ACH Return 的处理流程和用户通知完全未设计

**Wire 技术细节**:
- Wire 不可撤回（irrevocable），比 ACH 更安全
- 但用户需要主动从银行端发起 wire（不是从 app 内触发）
- **设计问题**: v3 入金页面"来源银行卡"的选择暗示用户从 app 端发起入金，但 wire 通常需要用户登录网银手动汇款。入金流程应该区分：(1) ACH = app 内发起；(2) Wire = 展示收款信息，用户去网银操作

### 12.2 HKD 入金

**设计文档现状**: v3 入金页面有币种切换（USD/HKD），但没有展示 HKD 特定的入金方式。

| 方式 | 需要支持 | 说明 |
|------|---------|------|
| FPS (转数快) | **是 — MVP 必需** | 即时到账，免手续费，HK 市场标配 |
| Bank Transfer (银行转账) | 是 | 传统银行间转账，1-2 工作日 |
| eDDA (电子直接扣款授权) | 建议支持 | 类似 ACH，从用户银行账户扣款 |
| Cheque (支票) | 可选 | 到账慢 (5-10 工作日) |

**严重缺失**: HKD 入金方式完全未设计。FPS 是香港零售金融的基础设施，日常转账几乎都用 FPS。如果不支持 FPS，HK 用户体验会很差。

**FPS 对接需求**:
- FPS 即时结算，到账后可立即用于交易
- FPS 需要提供 FPS ID 或二维码给用户
- FPS 有单笔限额和每日限额（各银行不同，通常单笔 HK$200,000-1,000,000）
- FPS 入金需要实时匹配（通过 FPS reference number 与用户账户关联）

### 12.3 即时入金 (平台垫资) 可行性分析

**设计文档现状**: v3 入金页面底部提到 "即时入金额度: $5,000。入金到账前，$5,000 可立即用于交易 (平台垫资)"。

**可行性评估**:

| 维度 | 分析 |
|------|------|
| **商业逻辑** | 平台先用自有资金给用户垫付，等 ACH 到账后收回。目的是提升用户体验（ACH 3-5天太慢）|
| **信用风险** | 用户银行账户余额不足导致 ACH Return → 平台损失垫付资金。**需要信用评估模型** |
| **资金成本** | 平台需要预备一定规模的资金池。如果 10,000 用户同时入金 $5,000 = $50,000,000 垫资规模 |
| **监管风险** | 平台垫资本质上是短期信贷行为，可能触发额外的监管要求 (lending license?) |
| **用户限制** | 垫付资金只能用于交易，不能出金。用户用垫付资金买入的股票，在 ACH 到账前不能卖出出金 |
| **Clawback 机制** | ACH Return 时，需要从用户账户扣回垫付金额。如果用户已用垫付资金买入且亏损，出现负余额 |

**建议**:

1. MVP 阶段可以支持即时入金，但需要严格的风控：
   - 仅对已完成至少一次成功 ACH 入金的用户开放
   - 额度分级：新用户 $1,000，活跃用户逐步提升到 $5,000-$10,000
   - 信用评分：基于历史入金成功率、账户年龄、交易行为
2. 垫付资金的使用限制必须在 UI 中明确标注：
   - "垫付资金仅可用于交易，不可出金"
   - "如入金未成功到账，垫付额度将被扣回"
3. **设计文档需要增加**: 即时入金失败 (ACH Return) 后的用户通知和处理流程

## 13. 出金审核流程

### 13.1 设计文档 vs 合规规则对照

**fund-transfer-compliance Rule 5 定义的三级审核**:

| 级别 | 合规规则要求的条件 | v3 设计是否覆盖 |
|------|------------------|---------------|
| **Auto-Approve** | 金额 <= 每日限额 AND 银行卡验证 >3天 AND 无 AML flag AND 非首次出金 AND 风险 LOW | **部分** — 设计有额度信息但未展示具体 auto-approve 逻辑 |
| **Manual Review** | 金额 >$50K OR 日累计 >80% 限额 OR 银行卡 <7天 OR 账户 <30天 OR 风险 MEDIUM/HIGH | **部分** — 出金拦截弹窗展示了 >$50K 触发审核，但其他条件未设计 |
| **Escalation** | 金额 >$200K OR SAR 触发 OR 多次 AML 失败 OR 内部 watchlist | **未覆盖** |

**差距分析**:

1. **冷却期判断逻辑**: v3 银行卡管理页展示了"冷却期 (还剩2天)"，很好。但 compliance rule 中有两个冷却期：3天 (auto-approve 条件) 和 7天 (manual review 触发条件)。设计只展示了一个。建议：绑定 0-3 天 = 不可出金，3-7 天 = 可出金但需人工审核，>7 天 = 正常。

2. **累计日出金限额**: v3 资金管理首页展示了"日限额: $100,000 (已用 $0)"，但出金页面没有引用这个数据。如果用户今天已出金 $80,000 (达到 80% 限额)，再发起出金应该触发 manual review。出金页面应实时展示"今日已出金额度"。

3. **首次出金标识**: compliance rule 要求首次出金需要 manual review，但设计中没有体现。建议在首次出金时增加额外验证步骤（如邮箱 + 短信双重验证码）。

4. **Escalation 场景**: 大额出金 (>$200K) 需要升级到合规官。设计中没有这个层级的审核状态。建议在出金状态时间轴中增加"合规审核中"步骤。

### 13.2 出金结算感知

v3 出金页面正确展示了可出金余额的计算：

```
可出金 = 总余额 - 未结算资金 - 待出金 - 保证金冻结
```

**补充要求**:

1. **即时入金垫付资金不可出金**: 如果用户使用了即时入金 $5,000，这部分在 ACH 实际到账前不能出金。可出金余额应再扣除"未到账的即时入金额度"。
2. **Pending buy orders**: 已挂的买单冻结的资金也应从可出金余额中扣除。v3 设计列了"保证金冻结"但未提及"挂单冻结"。
3. **当日卖出款**: 美股 T+1 结算，今天卖出的款项明天才结算。如果今天卖出 $10,000 AAPL，这 $10,000 在明天前是未结算状态，不能出金。v3 已经覆盖了"未结算资金"扣除，但需要确认实现中正确追踪每笔卖出的结算日期。

## 14. 银行账户验证

### 14.1 设计文档评估

v3 添加银行卡流程 (2.3) 设计了三步：选择银行 → 输入账户信息 → 安全验证 (Plaid / 小额打款)。

**正面评价**:
- Plaid 即时验证 + 小额打款双通道设计合理
- 账户持有人姓名自动填充 KYC 姓名
- "必须与 KYC 认证姓名一致"提示

**需要补充的技术细节**:

1. **Plaid 对接**:
   - Plaid Link SDK 集成到 app 中
   - Plaid 验证的是美国银行账户，HKD 账户不支持 Plaid
   - Plaid 返回 account holder name，后端必须与 KYC 姓名做 fuzzy matching（考虑 name order、middle name、abbreviation）
   - 同名校验失败时的 UI：设计未涉及。建议："银行账户姓名与您的认证姓名不一致，请确认是否为本人账户。如有疑问请联系客服。"

2. **小额打款验证**:
   - 发送 2 笔 $0.01-$0.99 的金额到用户银行账户
   - 用户在 1-3 工作日后回来输入金额进行验证
   - **设计缺失**: 用户回来输入金额的验证页面。用户在银行卡管理页看到"待验证"状态的卡，点击后应进入金额输入页面
   - 验证尝试次数限制：3 次输错 → 重新发起小额打款
   - 小额打款的回收：验证完成后，这两笔小额打款如何处理？通常是 ACH debit 回收，或直接作为 "welcome bonus"

3. **HKD 银行账户验证**:
   - 香港银行不支持 Plaid
   - 小额打款在 HK 也可行，但通过 FPS 发送更快（即时到账）
   - **设计缺失**: HKD 银行卡的验证方式。建议：(1) FPS 小额验证（即时）；(2) 上传银行月结单（显示账户姓名和号码）

4. **Cool-down 期**:
   - v3 展示了 3 天冷却期，合规规则也是 3 天
   - 冷却期计算：从验证完成时间开始，不是从提交绑卡时间开始
   - **设计确认**: v3 银行卡展示"绑定时间: 2026-03-01"和"冷却期 (还剩2天)"，这里的冷却期起算点需要确认

## 15. 双币种账户与换汇

### 15.1 设计文档现状

v3 资金管理首页 (2.1) 展示了 USD/HKD 双币种余额，换汇页面 (2.8) 设计了完整的换汇流程。

### 15.2 技术评估

**换汇流程**:

| 设计元素 | 评估 |
|---------|------|
| 实时汇率展示 + 15 秒刷新 | 合理。但需要明确：刷新期间用户确认的汇率是否锁定？还是以服务端处理时的汇率为准？ |
| 换汇手续费 0.3% | 合理。但手续费是否内含在汇率中 (spread) 还是单独收取？v3 同时展示了"参考汇率"和"实际汇率"（含手续费），这种透明展示是好的 |
| 滑动确认 | 合理。换汇是不可逆操作，需要明确确认 |
| 买卖方向切换 | 合理 |

**需要补充**:

1. **汇率锁定窗口**: 用户看到汇率并准备确认的这 15 秒内，汇率可能变动。需要一个汇率锁定机制 — 用户点击确认后，锁定当前汇率 X 秒（如 10 秒），在锁定期内完成交易。如果锁定期过期（如用户点了确认但 Face ID 验证太慢），需要重新获取汇率。

2. **最小换汇金额**: 未设定。建议最小 $10 USD 或 HK$100，避免小额换汇产生不成比例的系统开销。

3. **换汇与交易的联动**: 如果用户 USD 不足以购买美股但 HKD 充足，是否支持下单时自动换汇？如果支持，涉及两步操作的原子性问题（换汇成功但下单失败 → 需要回滚换汇？）。**建议 MVP 不支持自动换汇**，要求用户先手动换汇再下单。

4. **FX Rate 数据源**: 汇率从哪里获取？银行间汇率？彭博/路透？需要确认数据源的可靠性和更新频率。

5. **换汇的账务处理**: 一笔换汇在 ledger 中是两笔记录：USD 账户 debit + HKD 账户 credit（或反向）。需要保证两笔记录的原子性。

## 16. 账务系统设计

### 16.1 复式记账要求

**fund-transfer-compliance Rule 6** 要求：

1. **双向记账 (Double-entry)**: 每笔资金流动都有对应的借方和贷方
2. **Append-only**: 不可修改、不可删除
3. **Sum invariant**: 所有用户余额之和 = 平台托管账户余额
4. **每日对账**: 三方对账（内部账本、银行流水、托管方余额）

**v3 设计对账务系统的隐含需求**:

| 资金操作 | Debit 账户 | Credit 账户 | 备注 |
|---------|-----------|------------|------|
| ACH 入金到账 | 平台托管银行账户 | 用户 USD 账户 | -- |
| Wire 入金到账 | 平台托管银行账户 | 用户 USD 账户 | -- |
| ACH 出金 | 用户 USD 账户 | 平台托管银行账户 | -- |
| 买入成交 | 用户 USD 账户 | 用户证券持仓 | 金额 = 成交额 + 费用 |
| 卖出成交 | 用户证券持仓 | 用户 USD 账户 | 金额 = 成交额 - 费用 |
| 换汇 | 用户 USD 账户 | 用户 HKD 账户 | 两笔记录 |
| 即时入金垫资 | 平台垫资账户 | 用户 USD 账户 | -- |
| ACH 到账后回收垫资 | 用户 USD 账户 | 平台垫资账户 | -- |
| ACH Return (入金退回) | 用户 USD 账户 | 平台托管银行账户 | **需要处理余额不足的情况** |
| 佣金/费用 | 用户 USD/HKD 账户 | 平台收入账户 | -- |

**设计文档需要补充的前端视角**:

1. **余额明细页面**: v3 出金页面展示了"总余额"和"可出金余额"的分解。但缺少一个"余额明细"或"资金流水"页面，展示每一笔记账记录（类似银行的交易记录）。这与"出入金记录"不完全相同 — 出入金记录只记录入金/出金操作，但资金流水还包括交易产生的扣款/到账、费用扣收、利息（如有）等。

2. **对账异常通知**: 如果每日对账发现差异（>$0.01 阈值），需要通知运营团队。这是后台系统需求，但如果差异影响到用户余额，需要有用户端通知机制。

## 17. 即时入金风控

### 17.1 信用风险评估模型

v3 只是简单展示"即时入金额度: $5,000"，没有设计额度管理的逻辑。

**出入金引擎需要实现的风控模型**:

| 评估维度 | 权重 | 说明 |
|---------|------|------|
| 历史 ACH 成功率 | 高 | 过去 6 个月的 ACH 成功率。有 ACH Return 历史的用户降低额度 |
| 账户年龄 | 中 | 新账户 (<30天) 不开放即时入金或仅 $500 |
| KYC 等级 | 中 | Tier 1 (基础KYC) = $1,000, Tier 2 (增强KYC) = $5,000 |
| 信用评分 (外部) | 中 | 可选 — 如果获取了用户的信用报告数据 |
| 当前持仓市值 | 低 | 有稳定持仓的用户风险较低 |
| 过去 30 天出金模式 | 高 | 频繁入金后立即出金 → 疑似欺诈，禁止即时入金 |

### 17.2 Clawback 机制

当 ACH Return 导致即时入金垫资无法收回时：

```
场景: 用户即时入金 $5,000 → 买入 AAPL 28 股 → AAPL 下跌 → ACH Return
```

**处理流程** (需要 UI 设计):

1. ACH Return 确认后，从用户账户扣回 $5,000
2. 如果用户现金余额 >= $5,000 → 直接扣款，通知用户
3. 如果用户现金余额 < $5,000 → 产生负余额 (debit balance)
4. 负余额处理：
   - 冻结账户交易功能
   - 限制出金
   - 发送通知"您的账户存在 -$X.XX 欠款，请在 X 天内补足"
   - 逾期未补足 → 强制卖出持仓偿还
5. **设计需要增加**: 负余额状态页面、欠款通知、补缴入口

## 18. AML / 制裁筛查

### 18.1 OFAC SDN 筛查

**触发时机**:

| 操作 | 是否需要筛查 | v3 设计体现 |
|------|------------|-----------|
| 开户 (KYC) | 是 | 未明确体现，但 KYC 后台应包含 |
| 绑定银行卡 | 是 — 筛查银行名称、SWIFT code、国家 | **未体现** |
| 入金 | 是 — 筛查来源账户 | **未体现** |
| 出金 | 是 — 筛查目标账户 | 出金时间轴有"合规审核" |
| 换汇 | 是 | **未体现** |

**设计影响**:
- 每次 AML 筛查都可能产生 PENDING 状态（等待结果）或 BLOCKED 状态（命中制裁名单）
- 需要在入金和换汇流程中增加"合规审核中"状态
- AML BLOCKED 的用户通知："由于合规原因，您的操作无法完成。如有疑问请联系客服" — 不能告知用户命中了制裁名单

### 18.2 CTR (Currency Transaction Report)

**合规要求**: 单日入金或出金 > $10,000 USD 或 HK$120,000 时自动申报 CTR

**设计影响**:
- 不需要明确告知用户 "我们正在向 FinCEN 申报 CTR"（CTR 申报不要求告知用户）
- 但操作上可能增加额外的身份验证步骤
- **Structuring Detection**: 如果用户在 24 小时内多次入金且单笔 < $10,000 但总计 > $10,000（如 3 x $3,500），后台需要触发 structuring alert。这不阻断操作，但生成 SAR (Suspicious Activity Report) 供合规团队审查

### 18.3 Travel Rule

**合规要求**: Wire 转账 > $3,000 USD 或 > HK$8,000 时，需要传输 originator 和 beneficiary 信息。

**设计影响**:
- Wire 入金时，平台需要收集汇款人信息（通常银行已包含在 SWIFT 消息中）
- Wire 出金时，平台需要在 SWIFT 消息中包含完整的 originator 和 beneficiary 信息
- **v3 设计缺失**: 入金流程中没有收集 originator 信息的步骤。对于 Wire 入金，可以由后端从银行 SWIFT 消息中提取，不需要用户填写。但需要在后端验证信息完整性。

## 19. 幂等性与容错

### 19.1 前端防重复提交

v3 设计中的确认操作 (滑动确认 + Face ID) 在交互上已经降低了重复提交风险。但在以下场景仍需技术保障：

| 场景 | 风险 | 解决方案 |
|------|------|---------|
| 用户快速双击确认按钮 | 提交两次相同的入金/出金 | 前端 debounce + 按钮禁用 |
| 网络超时后用户重试 | 第一次请求可能已经成功 | **idempotency key** — 前端生成 UUID，重试时使用相同 key |
| App crash 后重启 | 不知道前次操作是否成功 | 重启后查询最近操作状态，提示用户 |
| 银行回调重复 | 同一笔入金银行回调两次 | 后端用银行 reference number 去重 |

**设计文档需要补充**:
- 网络超时后的 UI 处理：不是立即显示"操作失败"，而是"操作处理中，请稍候查看记录" + 自动跳转到记录页
- 关键操作提交后，按钮文案从"确认入金"变为"处理中..."，并禁用

### 19.2 银行通道超时处理

**fund-transfer-compliance Rule 10**: "Bank timeout: Do NOT assume success or failure; mark as PENDING and reconcile via bank statement"

**出入金引擎的 PENDING 状态管理**:

```
正常流程:  SUBMITTED → PROCESSING → COMPLETED / FAILED
超时流程:  SUBMITTED → PROCESSING → TIMEOUT_PENDING
                                        ↓
                                  (T+1 对账后)
                                    ↓          ↓
                              COMPLETED     FAILED
```

**设计影响**:
- 出入金记录页需要展示 TIMEOUT_PENDING 状态："处理中，预计 1 个工作日内确认"
- 不能因为超时就自动重试（可能导致重复扣款/入账）
- 用户如果看到 TIMEOUT_PENDING 并联系客服，客服应能查询银行通道状态

### 19.3 补偿交易 (Compensating Transactions)

当入金被银行退回 (ACH Return / Wire Return) 时：

1. 系统不能直接修改原始入金记录的状态（append-only 原则）
2. 正确做法：保留原始入金记录状态为 COMPLETED，新增一条 REVERSAL 记录
3. REVERSAL 记录从用户账户扣回入金金额
4. **设计需要增加**: 入金记录中的"已退回"状态标签，以及关联的退回记录

| 模块 | 估算工期 | 风险等级 | 说明 |

| FPS 对接 (HKD) | 3 周 | 高 | 需要 HK 银行的 FPS API 对接 |

| 出金审核流程 (三级审批) | 2 周 | 中 | 工作流引擎 |
| 换汇模块 | 2 周 | 中 | FX rate 源接入 + 汇率锁定 |

| **合计** | **~33 周 (8-9 个月)** | | 3-4 名后端工程师 |

**最高风险模块**:
1. **账务系统**: 复式记账的正确性是整个平台的基础。任何 bug 都可能导致用户余额不一致，引发监管风险。建议由最资深的工程师负责。
2. **ACH 通道对接**: 依赖外部银行/Plaid 的 production 环境，timeline 不完全可控。ACH Return 的各种 reason code (R01-R99) 都需要处理。
3. **即时入金**: 涉及信用风险，clawback 失败的场景处理复杂。建议作为 MVP 后续迭代功能，不放在首发版本。

### 20.3 MVP 分阶段建议

**Phase 1 (Month 1-5): 基础交易 + 基础出入金**
- 交易: 市价单 + 限价单，美股 only，单一 venue 路由
- 出入金: Wire 入金 + ACH 出金，USD only
- 目标: 能完成最基本的"入金 → 买卖美股 → 出金"闭环

**Phase 2 (Month 5-8): 完整交易 + 港股 + HKD**
- 交易: 所有订单类型，港股支持，SOR
- 出入金: ACH 入金 + FPS (HKD) + 换汇
- 目标: 美港股双市场全功能

**Phase 3 (Month 8-10): 高级功能**
- 即时入金 (垫资)
- 保证金交易
- Level 2 行情
- 高级风控 (position limits, concentration check)

**评审结论**:

设计文档 v1-v3 在移动端 UI/UX 层面完成度较高，尤其是 v3 补充的交易页面和出入金页面在信息展示、交互流程方面做得不错。但从交易引擎和出入金工程的视角来看，存在以下系统性差距：

1. **合规硬性要求的软化处理**: PDT 限制做成了可关闭的警告、AML 筛查在入金流程中缺失 — 这些是监管红线，不能妥协。
2. **后端复杂性在前端的投射不足**: 订单状态机的中间态、FIX 连接状态、银行通道超时 — 这些后端复杂场景都需要前端对应的状态展示和错误处理。
3. **港股规则覆盖不完整**: 开市前竞价时段、动态 board lot、VCM 都未涉及。港股不是"美股换个币种"，交易规则有根本性差异。
4. **出入金的 HKD 通道空白**: FPS 是 HK 市场的基础设施，必须支持。
5. **容错和异常处理需要系统性设计**: 不只是一张错误提示表，而是每个操作在超时、中断、并发、回退时的完整行为定义。

建议产品经理和 UX 设计师根据本评审报告的追踪表，按 CRITICAL > HIGH > MEDIUM 顺序补充设计，尤其是 CRITICAL 级别的 6 项（T-01, T-04, F-01, F-02 以及隐含的合规要求），必须在开发启动前完成设计闭环。
