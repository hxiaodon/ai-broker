# 技术部统一意见稿

> **文档类型**: 技术部门对产品设计方案的综合评审意见
> **评审范围**: mobile-app-design v1 / v2 / v3-supplement + UX 设计师审核问题清单 + HTML 原型
> **参与评审角色**: iOS 工程师、Android 工程师、前端工程师（Admin Panel）、后端工程师、交易引擎工程师、出入金工程师
> **日期**: 2026-03-11
> **状态**: 待产品经理逐项确认

---

## 一、总体评价

设计方案 v1→v2→v3 在移动端 UI/UX 层面完成度较高，核心交易链路（行情 → 下单 → 持仓 → 出入金）的页面设计清晰，v3 补充的详细线框图（登录/KYC、出入金、交易下单）质量良好。UX 设计师的评审问题清单（`design-review-for-pm.md`）也精准指出了信息架构不一致、港股细节缺失等问题。

但从**工程实现和合规落地**的角度，我们发现以下系统性问题：

1. **合规要求存在"软化"风险** — 部分监管硬性要求被设计为可选开关或仅做提示
2. **后端复杂性在前端的投射不足** — 状态机中间态、异常流程、容错处理缺少对应 UI
3. **港股不是"美股换个币种"** — 交易规则有根本性差异，设计覆盖严重不足
4. **Admin Panel 完全缺失** — 没有运营后台，无法合规运营
5. **多处文档不一致** — Tab 结构、订单状态枚举、KYC 步骤在不同版本间存在矛盾

**结论**: 技术可行，无根本性障碍。但有 **6 项 CRITICAL 级别问题**必须在开发启动前解决，否则会导致合规风险或重大返工。

## 二、CRITICAL 级别问题（开发阻塞项）

以下问题**必须在开发启动前由产品经理确认/修正**，否则技术部无法启动对应模块开发。

### CRITICAL-1: PDT 限制不能做成可关闭的开关

**问题**: v3 交易设置页有"显示PDT警告"开关，暗示用户可以关闭 PDT 限制提示。

**合规要求**: FINRA 强制要求 — Pattern Day Trader 规则（滚动 5 个工作日内 ≥4 次日内交易，且账户权益 <$25,000）必须**硬拦截**，不是可选警告。

**要求**: 删除"显示PDT警告"开关。交易引擎在下单前硬检查，不通过则直接拒绝。UI 保留 PDT 规则说明入口但不能有关闭拦截的开关。

### CRITICAL-2: Margin Call 设计完全缺失

**问题**: v3 有"账户类型选择（现金/保证金账户）"，但保证金账户的 Margin Call 通知、强制平仓流程、维持保证金监控页面完全没有设计。

**要求**: 如果 MVP 支持保证金账户，必须补充 Margin Call 完整流程设计；如果 MVP 仅支持现金账户，请明确确认并从 KYC 流程中移除账户类型选择步骤。

### CRITICAL-3: 入金流程缺少 AML 筛查

**问题**: v3 入金页面没有 AML/OFAC 筛查环节。按照合规规则 Rule 2，**每笔资金转移都必须通过 AML 筛查**。

**要求**: 入金确认后、资金到账前增加"合规审核中"状态。大多数筛查可在后台秒级完成（不影响用户体验），但筛查结果为 REVIEW 或 BLOCKED 时需要对应的用户通知和状态展示。

### CRITICAL-4: Travel Rule 信息收集缺失

**问题**: Wire 转账 > $3,000 USD 或 > HK$8,000 时，合规规则 Rule 3 要求传输 originator/beneficiary 完整信息。v3 设计中没有收集这些信息的步骤。

**要求**: Wire 入金 — 后端从银行 SWIFT 消息提取（无需用户填写），但后端必须验证完整性。Wire 出金 — 确认 KYC 信息是否满足 originator 信息要求，不足的需要补充收集。

### CRITICAL-5: Admin Panel 必须与移动端同步立项

**问题**: 三版设计文档**完全没有涵盖运营后台**。KYC 人工审核、出金审批、AML 筛查结果处理、订单监控、CTR/SAR 申报 — 这些全部需要 Admin Panel 支撑。

**要求**: Admin Panel 不是"后续补充"的附加功能，而是合规运营的基础设施。产品需为以下模块补充需求：
- KYC 审核后台（审核队列、证件查看器、审批操作）
- 出金审核工作台（三级审批流程）
- 订单监控与异常告警
- 合规报告中心（CTR/SAR 管理）
- 用户账户管理
- 系统配置管理

**前端工程师估算**: Admin Panel MVP 约 **16-20 周（4-5 个月）**，需与移动端并行开发。

### CRITICAL-6: KYC SDK 选型须立即启动

**问题**: KYC 人脸识别 + 活体检测依赖第三方商用 SDK（Jumio/Onfido/阿里云/腾讯云），商务合同签署 + 技术评估周期需 **4-8 周**。

**要求**: 本周内启动 SDK 评估和供应商接洽。建议评估维度：
- 证件覆盖范围（中国身份证 + 香港 HKID + 美国驾照/护照）
- 活体检测精度和合规认证
- iOS/Android SDK 包体积影响（目标 <15MB）
- 定价和合同周期

## 四、各角色关键发现汇总

### 4.1 iOS 工程师

| 维度 | 关键发现 |
|------|---------|
| 最高风险模块 | KYC 人脸识别 SDK 集成、K 线图自定义渲染、Plaid SDK 集成 |
| 图表库建议 | **不推荐** danielgindi/Charts（UIKit 架构，SwiftUI 集成摩擦大），建议评估 TradingView LightweightCharts iOS SDK |
| WebSocket 建议 | 优先用 `URLSessionWebSocketTask`（iOS 13+），无需引入 Starscream |
| 网络层建议 | 可不引入 Alamofire，直接 `URLSession` + `async/await` + `CryptoKit` HMAC 签名 |
| Plaid 限制 | **Plaid 不支持 HK 银行**，HKD 账户只能走小额打款验证 |
| 平台增值 | 建议 P1 增加 Live Activities（灵动岛订单状态）、WidgetKit（自选股小组件） |
| MVP 工期 | 单人 ~35 周（9 个月），双人并行约 5-6 个月 |

### 4.2 Android 工程师

| 维度 | 关键发现 |
|------|---------|
| 最高风险模块 | KYC 活体检测 SDK、K 线图手势交互、华为 HMS 推送适配 |
| 图表库建议 | MVP 使用 MPAndroidChart（`AndroidView` 包装），P1 考虑 Compose Canvas 自定义替换 |
| 设备碎片化 | MIUI/ColorOS 后台杀进程影响 WebSocket；华为无 GMS 需集成 HMS Push Kit |
| 性能方案 | LazyColumn 按 symbol 分离 StateFlow + 100ms 节流，BigDecimal 计算在后台协程 |
| 安全补充 | 建议下单页和确认流程全程 `FLAG_SECURE`（屏蔽截图），不仅仅是弹窗 |
| 离线设计缺失 | 当前设计未体现离线状态 UI — 需补充网络断开/WebSocket 断连的提示和降级方案 |
| MVP 工期 | 双人并行约 10-12 周（含联调测试） |

### 4.3 前端工程师（Admin Panel）

| 维度 | 关键发现 |
|------|---------|
| 核心结论 | Admin Panel 是合规运营基础设施，**不能延后**，需与移动端同步立项 |
| 模块规模 | 7 大模块、约 40 个页面 |
| P0 模块 | KYC 审核、出入金管理、订单监控、用户管理、合规报告中心（均为监管强制） |
| 一致性风险 | 订单状态枚举 v1/v2 不统一、KYC Tier 未定义、Risk Score 计算规则未说明 |
| 缺失的运营支撑 | 人工视频审核系统、Plaid 异常处理后台、W-8BEN 续签管理、即时入金风控仪表板 |
| MVP 工期 | **16-20 周（4-5 个月）** |

### 4.4 后端工程师

| 维度 | 关键发现 |
|------|---------|
| 协议选型 | 移动端 → REST Gateway（grpc-gateway），内部服务 → gRPC，实时数据 → WebSocket |
| 服务划分 | API Gateway + Auth + Account + KYC/AML + Notification + Reporting（6 个服务） |
| 核心 API | 梳理出 40+ 个 API 端点，建议尽快启动 API Contract 设计以解除前端阻塞 |
| 数据库要点 | 金额全部 `NUMERIC(20,4)`，PII 应用层 AES-256-GCM 加密，审计日志 Row Security Policy 禁止修改 |
| 游客模式 | 双轨缓存（实时 + 15 分钟延迟），响应头 `X-Data-Delay` 标识 |
| 最高风险 | KYC 第三方集成（Onfido/Jumio）合同谈判 + 接入审批需 4-8 周 |
| MVP 工期 | ~54 人周（后端服务部分，不含交易引擎和出入金） |

### 4.5 交易引擎工程师

| 维度 | 关键发现 |
|------|---------|
| 订单状态机 | 缺少 CANCEL_REQUESTED、REPLACE_REQUESTED、EXPIRED 中间态及对应 UI |
| PDT 规则 | **严重问题** — "显示PDT警告"开关必须删除，PDT 是 FINRA 强制要求的硬拦截 |
| 港股缺失 | 开市前竞价时段（09:00-09:30）未设计，Board Lot 不能硬编码 100 股 |
| 熔断缺失 | 美股 LULD（Limit Up-Limit Down）和港股 VCM（波幅调节机制）都未涉及 |
| Best Execution | 需补充 Rule 606 执行质量报告、PFOF 披露，市价单需价格保护（collar 机制） |
| FIX Protocol | MVP 最高风险模块，交易所 UAT 排期长，certification 流程耗时不可控 |
| MVP 工期 | ~29 周（7-8 个月），4-5 名后端工程师 |

### 4.6 出入金工程师

| 维度 | 关键发现 |
|------|---------|
| HKD 通道空白 | FPS（转数快）完全未设计 — FPS 是 HK 零售市场基础设施 |
| Wire 入金误解 | v3 设计将 Wire 入金设计为"从 App 发起"，实际应展示平台收款信息由用户从银行端发起 |
| ACH Return | 入金退回的用户通知流程和负余额处理完全缺失 |
| 即时入金风控 | 垫资机制缺少信用评估模型、Clawback 失败处理、负余额页面 |
| 账务系统 | 需要完整的复式记账（Double-entry Ledger），是整个平台的基石，需最资深工程师负责 |
| 合规覆盖 | 10 条合规规则中，入金 AML 筛查（Rule 2）、Travel Rule（Rule 3）、三级审批（Rule 5）覆盖不足 |
| MVP 工期 | ~33 周（8-9 个月），3-4 名后端工程师 |

## 五、统一的 MVP 分阶段建议

综合各角色意见，建议将 MVP 分为 3 个 Phase：

### Phase 1: 核心闭环（Month 1-5）

**目标**: 能完成"注册 → KYC → 入金 → 买卖美股 → 出金"完整闭环

| 模块 | 范围 | 负责角色 |
|------|------|---------|
| 移动端 | 登录注册、KYC（简化版：人工审核替代活体检测）、行情、交易（市价单+限价单）、持仓、订单 | iOS + Android |
| 交易引擎 | 美股 only、单一交易所路由、基础风控（购买力+PDT） | 交易引擎 |
| 出入金 | Wire 入金 + ACH 出金、USD only、银行卡管理 | 出入金 |
| 后端 | Auth + Account + KYC（基础版） + Notification + 行情 Gateway | 后端 |
| Admin Panel | KYC 审核 + 出金审批 + 用户管理（最小可用版） | 前端 |

### Phase 2: 双市场 + 完整功能（Month 5-8）

| 模块 | 新增范围 |
|------|---------|
| 港股交易 | Board Lot、Tick Size、竞价时段、T+2 结算 |
| HKD 出入金 | FPS 入金、换汇、双币种账户 |
| 完整订单类型 | 止损、止损限价、MOO/MOC、IOC/FOK |
| KYC 完整版 | OCR SDK + 活体检测 + 手写签名 |
| Admin Panel 增强 | 订单监控、合规报告、对账系统 |

### Phase 3: 高级功能（Month 8-10）

| 模块 | 新增范围 |
|------|---------|
| 即时入金（垫资） | 信用评估 + Clawback |
| 保证金交易 | Margin Call + 维持保证金监控 |
| 高级行情 | Level 2 深度数据、技术指标（MACD/RSI/BOLL） |
| 平台增值 | Live Activities、Widgets、价格提醒 |
