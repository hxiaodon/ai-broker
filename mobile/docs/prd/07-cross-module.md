## 四、行情 → 交易

### 4.1 导航参数传递

从行情/股票详情页进入交易委托页时传递参数：

| 参数 | 来源 | 用途 |
|------|------|------|
| symbol | 当前股票代码 | 委托标的预填 |
| side | 点击"买入"→BUY / "卖出"→SELL | 委托方向预填 |
| current_price | 行情 WebSocket 最新价 | 限价单默认价格参考 |
| market_status | 当前交易时段 | 决定可用订单类型 |

### 4.2 交易时段对订单类型的限制

| 时段 | 可用订单类型 |
|------|------------|
| 常规（Regular） | 市价单 + 限价单 |
| 盘前（Pre-Market） | 仅限价单（需用户确认风险） |
| 盘后（After-Hours） | 仅限价单（需用户确认风险） |
| 休市（Closed） | 禁止下单；可预设次日限价单（Phase 2） |
| 交易暂停（Halt） | 显示暂停原因，禁止下单 |

## 五、交易 → 持仓

### 5.1 成交后持仓更新流程

```
交易所成交确认（FIX 回报）
    ↓
交易引擎处理成交记录
    ↓
更新 positions 表（均价重算，持仓数量更新）
    ↓
写入 order_fills 表
    ↓
写入 ledger_entries 表（资金 debit/credit）
    ↓
Kafka 发布成交事件
    ↓
WebSocket 推送至客户端（持仓页实时刷新）
    ↓
推送通知：成交提醒
```

### 5.2 成交通知内容

| 情景 | 通知文案 |
|------|---------|
| 买入完全成交 | "✅ 已买入 AAPL 100 股，成交价 $182.52" |
| 卖出完全成交 | "✅ 已卖出 TSLA 50 股，成交价 $245.00，盈亏 +$500.00" |
| 部分成交 | "⚡ NVDA 委托部分成交：80/100 股 @ $650.00" |
| 委托被拒绝 | "❌ AAPL 委托被拒绝：资金不足" |
| 委托撤销 | "撤单成功：AAPL 限价委托 100 股 @ $180.00" |
| GTC 委托到期 | "⏰ AAPL GTC 委托（100 股 @ $180.00）已到期" |

### 5.3 持仓页与交易页的资金联动

**买入前检查**（前端本地校验，下单前实时计算）:
```
预计买入金额 = 委托数量 × 委托价格
实际可用资金 = 可用现金 − 已冻结委托金额
校验：预计买入金额 ≤ 实际可用资金
```

**"最大"数量计算**:
- 买入：`floor(实际可用资金 / 委托价格)`，市价单用 Ask 价估算
- 卖出：当前持仓中已结算的股数

## 六、出入金 → 持仓

### 6.1 入金完成 → 余额更新

```
入金状态变更为 COMPLETED
    ↓
写入 ledger_entries（credit 用户账户）
    ↓
更新用户可用余额
    ↓
WebSocket 推送余额变更至客户端
    ↓
推送通知："$1,000.00 已成功存入您的账户"
```

### 6.2 出金触发 → 余额冻结

```
用户提交出金申请
    ↓
立即冻结该金额（不可用于买入，不计入可提现余额）
    ↓
写入 ledger_entries（pending debit）
    ↓
后续：出金完成 → 正式 debit；出金失败 → 解冻
```

### 6.3 持仓页快速入金

从持仓页的"入金"快捷按钮到出入金页的路由参数：

```
navigation.push('/funding', {
  default_tab: 'deposit',
  source: 'portfolio_shortcut'
})
```

## 七、KYC → 出入金

### 7.1 KYC 名称用于银行卡同名验证

| 数据 | 来源 |
|------|------|
| 用户法定英文姓名 | KYC Step 1（first_name + last_name） |
| 银行卡持有人姓名 | 银行卡绑定页自动填充（只读） |

验证逻辑（服务端）：
1. 去除首尾空格，转大写
2. 允许模糊匹配：名字顺序（First Last vs Last First）
3. 允许中间名省略（例：ZHANG SAN vs ZHANG MINGSAN）
4. 若完全不匹配 → 拒绝绑卡，提示"银行账户持有人姓名与您的开户姓名不符"

### 7.2 W-8BEN → 股息税率

KYC 签署 W-8BEN 后，后台系统标记：
- 股息预扣税率：10%（协定税率）而非默认 30%
- 用于持仓中涉及股息收入的税务处理

### 7.3 KYC Tier → 出入金限额检查

出入金申请提交时，系统实时查询用户当前 KYC Tier 并校验：

```
检查：本次申请金额 ≤ 单笔限额（Tier）
检查：今日已操作金额 + 本次金额 ≤ 日限额（Tier）
检查：本月已操作金额 + 本次金额 ≤ 月限额（Tier）
```

限额不足时返回明确提示：
```
"本次入金金额超过您当前等级的单笔限额（$50,000）。
 请降低金额，或完善 KYC 材料提升账户等级。"
```

## 八、交易 → 出入金（资金结算联动）

### 8.1 T+1 结算流程

```
美股交易成交（Trade Date = T）
    ↓
结算日 = T + 1 个工作日（跳过周末和美国节假日）
    ↓
T+1 当日，托管行完成结算
    ↓
系统更新：
  - 买入：从 unsettled_positions 转移至 positions（settled_quantity +）
  - 卖出：从 unsettled funds 转入 available cash
    ↓
持仓页可卖数量更新
出入金页可提现余额更新
推送通知（如有）
```

### 8.2 结算日计算

```go
// 美国市场节假日需维护列表（纽交所节假日日历）
func calcSettleDate(tradeDate time.Time) time.Time {
    settle := tradeDate
    addedDays := 0
    for addedDays < 1 {
        settle = settle.AddDate(0, 0, 1)
        if isNYSEBusinessDay(settle) {
            addedDays++
        }
    }
    return settle
}
```

### 8.3 资金冻结 / 解冻生命周期

买入委托从创建到结算，涉及资金的完整生命周期如下：

```
阶段 1：委托创建（风控通过时）
  → ledger_entries: entry_type=HOLD, debit_amount=预估委托金额（数量×限价）
  → ledger_accounts.balance 不变，但 available_cash = balance - SUM(HOLD entries)

阶段 2a：委托撤单 / 拒绝 / 过期
  → ledger_entries: entry_type=RELEASE, credit_amount=HOLD金额（冲销）
  → available_cash 恢复

阶段 2b：委托全部成交（FILLED）
  → ledger_entries: entry_type=RELEASE（释放 HOLD）+ TRADE_BUY（实际扣款）
  → 若实际成交金额 < HOLD金额，差额通过 RELEASE 退回
  → ledger_accounts.balance 减少实际成交金额+费用

阶段 3：T+1 结算
  → unsettled_positions 转入 positions（settled_quantity 增加）
  → 卖出：entry_type=SETTLEMENT，credit_amount=卖出所得（计入 balance）
  → positions.version 通过乐观锁 +1（并发保护，见 PRD-06 Section 8）
```

**HOLD 金额计算规则**：
- 限价买入：冻结金额 = 限价 × 数量（+ 预估费用）
- 市价买入：冻结金额 = Ask × 数量 × 1.05（5% buffer，防止 Collar 触发后金额不足）

**注**：HOLD/RELEASE entry 不代表实际资金移动，仅用于计算可用余额（available_cash）。
实际资金只在 TRADE_BUY/TRADE_SELL/DEPOSIT/WITHDRAWAL 时变化。

## 九、全局异常状态

### 9.1 网络断开处理

| 状态 | 展示 |
|------|------|
| 网络断开 | 顶部 Banner："连接已断开，正在重连..." |
| 重连成功 | Banner 自动消失（3 秒） |
| 断网期间有委托 | "请前往订单列表确认委托状态" |

**断网期间行为**:
- 禁止新下委托（提交按钮灰化，提示"网络不可用"）
- 禁止提交出入金（提示确认后手动重试）
- 行情数据显示最后更新时间戳
- WebSocket 自动重连（指数退避，最长 30 秒间隔）

### 9.2 股票交易暂停（Trading Halt）

| 步骤 | 行为 |
|------|------|
| 检测到 Halt 信号 | 行情服务推送 Halt 状态 |
| 股票详情页 | 显示红色横幅："该股票交易暂时中止" + 暂停时间 + 预计恢复（若有）|
| 委托页 | 买入/卖出按钮禁用，显示暂停提示 |
| 恢复提醒 | 用户可设置"恢复时提醒"，系统检测到 Halt 解除后推送通知 |

### 9.3 行情数据中断

| 情景 | 处理 |
|------|------|
| WebSocket 断连 | 自动重连，显示最后更新时间 |
| 数据源故障 | 显示"行情数据暂时不可用，显示最近一次数据" |
| 超过 5 分钟无更新 | 界面提示，自动触发全量刷新 |

## 十二、用户数据一致性保障

### 12.1 读操作

- 余额、持仓、订单等关键数据**始终从主库（PostgreSQL）读取**，不接受 Redis 缓存数据作为唯一来源
- WebSocket 推送的行情数据用于 UI 展示，但费用计算、限额校验必须经后端 API 验证

### 12.2 写操作幂等性

所有涉及资金或订单的 POST 请求必须携带 `Idempotency-Key`（UUID v4），后端保证：
- 相同 Key 的重复请求只执行一次
- 72 小时内返回相同响应

### 12.3 乐观锁机制

余额更新使用乐观锁防止并发冲突：
```sql
UPDATE ledger_accounts
SET balance = balance + :delta,
    version = version + 1
WHERE user_id = :user_id
  AND version = :expected_version;
```

并发冲突（version 不匹配）→ 重试最多 3 次 → 仍失败则返回错误要求用户重试。
