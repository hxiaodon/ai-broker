### P2-8: 离线优先架构 (完整化)
**状态**: 部分完成 (2-3周工作量)

**问题**: P0-2 实现了缓存，但离线体验不完整

**当前完成**:
- ✅ Drift SQLite 缓存层
- ✅ API fetch-first → cache fallback
- ✅ 30s TTL 缓存策略

**缺失的部分**:
1. **离线 UI 指示器** (必须)
   - "离线模式"横幅
   - "数据可能延迟"提示
   - 连接状态显示
   - 工作量: 1 day

2. **数据同步队列** (重要)
   - 缓存用户的编辑操作（离线时）
   - 网络恢复后自动同步
   - 工作量: 2-3 days

3. **冲突解决策略** (重要)
   - 用户离线编辑 → 回来发现数据已变
   - Last-write-wins vs 提示用户选择
   - 工作量: 2 days

4. **后台同步** (可选)
   - 定期检查网络并同步
   - 省电模式下暂停
   - 工作量: 2-3 days

**优先级**: ⭐⭐⭐⭐ (高) — 影响用户体验

---

### P3-9: 代码生成自动化
**状态**: 待实施 (1-2周工作量)

**问题**: 手工重复代码（mappers、serializers）可自动化

**当前状态**:
- ✅ 已有 `freezed` 用于数据类
- ✅ 已有 `json_serializable` 用于 JSON
- ❌ 缺少 Riverpod 代码生成优化
- ❌ 缺少自定义 mapper 生成工具

**可以自动化的**:
1. Riverpod provider 的 freezed 集成
2. 自动生成 DTO → Entity mappers
3. 自动生成 Entity → UI state converters
4. 工作量: 1-2 days

**ROI**: 
- 减少重复代码 ~200-300 lines
- 减少维护负担（改字段自动更新映射）
- 相对工作量大，收益中等

**优先级**: ⭐⭐ (低) — 优化代码，非关键

---

### P3-10: 性能优化
**状态**: 待实施 (2-3周工作量)

**问题**: 无性能基准和优化目标

**当前状态**:
- ❓ 不知道主要性能瓶颈在哪
- ❓ 缺少性能监控和告警
- ❓ WebSocket 吞吐量未优化

**需要做**:
1. **性能基准测试** (2-3 days)
   - 市场数据加载时间
   - WebSocket 消息吞吐量
   - 本地缓存查询速度
   - 图表渲染性能

2. **识别瓶颈** (1 day)
   - 运行 Flutter DevTools profiler
   - 使用 Dart Observatory
   - 识别 CPU/内存热点

3. **优化目标设定** (1 day)
   - 首屏加载 < 2s
   - WebSocket 消息 < 100ms 延迟
   - 图表滑动 FPS > 50

4. **逐步优化** (1-2 weeks)
   - 代码分割和懒加载
   - 图表虚拟化
   - WebSocket 消息合并
   - 缓存预热策略

**ROI**: 
- 影响用户留存和满意度
- 需要对标竞品的性能指标
- 工作量大，但长期收益高

**优先级**: ⭐⭐⭐ (中) — 重要但可延后
