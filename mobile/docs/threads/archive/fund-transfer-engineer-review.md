## 详细问题分析

### 问题 1: 可提现余额公式符号矛盾 [P0 — 合规/资金安全]

**问题描述**

PRD-05 第 5.2 节的公式与注释直接矛盾：

- 公式写的是：`可提现余额 = 现金余额 − 未结算买入资金 + 未结算卖出资金`
- 注释写的是：「卖出产生的未结算资金也不可立即提现」

`+ 未结算卖出资金` 与注释逻辑完全相反。API 响应示例中 `unsettled_sells: "-300.00"`（负值），与公式加号矛盾，三处表达无一致口径。

**正确公式**

从 API 示例还原：`withdrawable_balance = 5000 - 500 - 300 = 4200`，即两项均为减法：

```
可提现余额 = total_cash
           − unsettled_buys（等待结算的买入占用）
           − unsettled_sells（卖出但未结算，不可提）
           − pending_withdrawals
```

**建议修复**

1. 公式第三行改为 `− 未结算卖出资金（已卖出但 T+1 未结算，不可提）`
2. API 响应字段 `unsettled_sells` 值改为正数（`"300.00"`），文档注明两项均做减法
3. 补充 `total_cash` 的精确定义（是否包含未结算卖出所得）

### 问题 2: 银行退汇（ACH Return）处理逻辑完全缺失 [P0 — 资金安全]

**问题描述**

PRD-05 第 5.4 节仅定义 `FAILED`（银行退汇）为终态，描述为"资金返还 + 推送通知"，但完全没有说明三个关键子问题：

**子问题 A — ACH Return Code 分类处理**

ACH 退汇有 80+ 种 Return Code，必须分类处理：

| Return Code 类别 | 含义 | 正确处理 |
|----------------|------|---------|
| R01（资金不足）| 用户余额不足 | 标记失败，通知用户 |
| R02（账户已关闭）| 银行账户注销 | 立即暂停银行卡，触发人工审核 |
| R07/R10（未授权）| 用户声称未授权 | **触发 SAR 流程**，冻结账户调查 |

R07/R10（未授权退汇）是高风险欺诈信号，PRD 完全未提及，存在合规盲区。

**子问题 B — 资金已入账后退汇（Late Return）**

ACH 退汇窗口为 2-5 个工作日，极端情况：入金 COMPLETED → 用户用这笔钱买入股票 → 银行 R07 退汇到来。此时用户余额不足以覆盖退汇金额，PRD 无处理逻辑。

**子问题 C — Ledger REVERSAL 条目设计缺失**

`ledger_entries.entry_type` 包含 `REVERSAL`，但没有说明 REVERSAL 条目结构规范，且无 `parent_entry_id` 字段关联原始条目。

**建议解决方案**

1. PRD 补充 ACH Return Code 处理矩阵，至少覆盖 R01-R10
2. 定义「退汇后余额不足」的 Saga 补偿流程
3. `ledger_entries` 表补充 `parent_entry_id UUID` 字段

### 问题 3: `ledger_accounts` 表缺失，乐观锁无处落地 [P0 — 数据完整性]

**问题描述**

PRD-07 第 12.3 节定义了乐观锁机制，引用了 `ledger_accounts` 表（含 `balance` + `version` 字段）。但 PRD-05 数据模型（第十节）**没有 `ledger_accounts` 或任何余额快照表**，只有只追加的 `ledger_entries`。两个 PRD 的账户余额存储方案相互矛盾。

**影响分析**

若余额仅靠 `ledger_entries` 聚合，高并发场景下无法用乐观锁防止超扣：两个并发出金请求都读到 `SUM=1000`，各自认为可以扣款 600，均成功插入 ledger 条目，实际合计扣除 1200，账户余额变为 -200。

**建议解决方案**（二选一，并在 PRD 中明确）

方案 A（推荐）：引入 `account_balances` 快照表（即 PRD-07 所引用的 `ledger_accounts`），与 `ledger_entries` 同一事务更新，乐观锁施加在快照表上。

方案 B：使用 PostgreSQL `SELECT ... FOR UPDATE` 行锁，去掉版本号字段，改为悲观锁策略。

### 问题 4: KYC Tier 1 入金权限与银行卡绑定前置条件矛盾 [P1 — 功能正确性]

**问题描述**

- PRD-07 定义 `PENDING_REVIEW` 状态可以出入金（Tier 1 限额）
- 但银行卡绑定需要「账户持有人姓名从 KYC 自动填充，只读」
- KYC `IN_PROGRESS` 状态时，Step 1 已填写但 Step 7 未提交，系统是否可读取 `first_name`？

更深层问题：若 OCR 识别失败，用户手动填写姓名但信息不实，系统在 KYC 未通过时就用这个姓名做同名验证，存在合规风险。

**建议解决方案**

明确 Tier 1 银行卡绑定的前置条件：必须是 `PENDING_REVIEW` 及以上（Step 7 全部提交后），`IN_PROGRESS` 不可绑卡。

### 问题 5: 冷却期规则自相矛盾 [P1 — 功能正确性]

**问题描述**

PRD-05 存在两条针对新绑银行卡的时间规则：

| 规则 | 时间点 | 处理 |
|------|-------|------|
| 冷却期（3.2 节）| 绑定后 0-3 天 | 无法出金（硬性拦截） |
| 人工审核触发（5.3 节）| 绑定后 < 7 天 | 转人工审核（非硬性拦截） |

在 3-7 天窗口期内：冷却期已结束，用户认为可以正常出金，但系统仍触发人工审核，用户体验混乱。两个规则的叠加关系 PRD 未定义。

**建议解决方案**

明确三个阶段：
- 绑定后 0-3 天：硬性拦截，显示"冷却期（剩余 X 天）"
- 绑定后 3-7 天：可发起出金，但强制转人工审核，用户提交前展示人工审核弹窗
- 绑定后 7 天以上：满足其他条件时可自动审批

### 问题 6: 银行回调幂等性缺失，存在双重入账风险 [P1 — 资金安全]

**问题描述**

`deposits` 表有 `idempotency_key UNIQUE`，用于防止客户端重复提交。但 ACH/Wire 到账回调来自银行，不受客户端 Idempotency-Key 控制。

`bank_reference` 字段（银行参考号）在 PRD 中没有说明是否做 UNIQUE 约束或显式去重处理。银行可能重发相同到账通知，或 Kafka 消费者崩溃重启后重消费同一条事件。

**建议解决方案**

1. `deposits` 表对 `bank_reference` 增加 UNIQUE 约束（非 null 值）
2. 明确 ACH 回调处理的幂等逻辑：先查 bank_reference，若已 COMPLETED 则直接返回
3. Wire 到账使用 IMAD 作为唯一幂等键，PRD 应明确

### 问题 8: 微存款验证暴力破解防护不足 [P1 — 安全]

**问题描述**

微存款金额范围：$0.01-$0.99，共 99 种可能；两笔组合：99 × 99 = **9,801 种**；允许失败 3 次；成功概率 ≈ 0.031%。

更关键的是：PRD 说"验证失败需重新绑定"，但没说重新绑定后是否重置尝试次数，若可以无限次重新绑定，3 次限制形同虚设。

**建议解决方案**

1. 同一用户对同一银行账号哈希的总尝试次数限制（独立于绑定次数，最多 5 次）
2. 增加指数退避锁定：第 3 次失败后锁定 24 小时，第 4 次升级人工审核
3. 增加 IP 级别频率限制，防止分布式爆破

### 问题 9: Structuring Detection 技术实现规格缺失 [P1 — 合规]

**问题描述**

PRD-05 第 6.2 节定义结构性交易检测，但存在关键歧义：
1. **24 小时窗口**：自然日还是滑动窗口？监管通常要求滑动窗口
2. **计数口径**：只计入金，还是双向合并计算？
3. **拦截时序**：同步拦截还是异步标记？若异步，前两笔已 COMPLETED 时如何追溯审查？

**建议解决方案**

补充技术规格：使用 Redis ZADD + ZRANGEBYSCORE 实现滑动窗口（性能优于实时 SQL 聚合），在 AML_SCREENING 阶段同步执行，入金和出金独立计算。

### 问题 10: Travel Rule ACH 实现字段映射模糊 [P2 — 合规可实现性]

**问题描述**

标准 Nacha ACH 格式的 Addenda Record 仅有 80 个字符，承载完整 Travel Rule 信息格式上极为勉强，且 Nacha 并无原生 Travel Rule 字段定义。FinCEN 对 ACH 的 Travel Rule 要求实际上是通过银行间内部记录保存实现，而非 ACH 报文本身。

**建议解决方案**

将描述修改为：出金 > $3,000 时在平台侧完整留存发/收款方信息，通过银行的 Supplemental Data 字段传递（各家银行 API 不同）。`withdrawals` 表补充 `travel_rule_data JSONB` 字段。

---

### 问题 11: 同名验证中文用户英文姓名边缘案例覆盖不足 [P2 — 合规]

**问题描述**

PRD 定义的模糊匹配规则仅覆盖三种情况，但中国大陆/香港用户与美国银行账户名差异远不止于此：

| 场景 | KYC 姓名（护照）| 美国银行账户名 | 能否匹配 |
|------|--------------|------------|--------|
| 拼音连字符 | ZHANG MINGSAN | ZHANG MING-SAN | 否 |
| 双名空格 | ZHANG MING SAN | ZHANG MINGSAN | 否 |
| 港式英文名顺序 | CHAN TAK WAI | TAKWAI CHAN | 部分匹配 |

**建议解决方案**

定义分级匹配策略：完全匹配 → 自动通过；Token Set 匹配度 ≥ 阈值 → 自动通过；其他 → 转人工复核。增加"姓名说明"豁免流程。

---

### 问题 12: 出金最低金额未定义 [P2 — 产品完整性]

**问题描述**

入金：ACH 最低 $100，Wire 最低 $1,000。出金：无任何最低金额限制，可能出现 $0.01 出金但产生 $25 Wire 手续费（净出金为负）。

**建议解决方案**

补充出金最低金额：ACH 出金最低 $1（或与入金对称），Wire 出金最低 $1,000。后端校验 `amount > fee`，不能仅依赖前端拦截。

## 附：Cross-cutting 问题

### 入金状态机缺少 `RETURNED` 终态

PRD-05 入金状态机只有 `DEPOSIT_BLOCKED`（合规拦截）作为失败终态，没有"银行处理失败"终态。建议补充 `RETURNED`（银行退汇）终态，区分于 `DEPOSIT_BLOCKED`（主动合规拒绝）。

### 对账规格缺失

验收标准提到"每日对账"但没有定义：对账任务触发时机、银行对账文件格式（NACHA 84-char format）、对账不平时的人工处理流程、微存款发出后的对账识别。建议增加"对账规格"章节。
